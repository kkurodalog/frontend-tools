## 1. 質問フロー機能テスト（9カテゴリ・二層構造）

| # | 観点 | 合格条件 |
|---|---|---|
| 1-1 | コア5問フローで完成（二層構造 §2-2） | ①画像タイプ・②被写体・③雰囲気・④光・⑥構図(位置)の5問だけで完成プロンプトが揃い、コピーできる |
| 1-2 | 「ここまでで完成／さらに詰める」区切り（§6-1） | コア5問完了で完成区切りが明示され、深掘りに進まなくても完成プロンプトが出る |
| 1-3 | 深掘り4段の段階投入（⑤質感/⑦不完全さ/⑧ネガティブ詳細/⑨アスペクト比） | 「詳しく詰める」を選んだ時のみ表示され、選んだ項目だけプロンプトに上乗せされる |
| 1-4 | 各カテゴリの選択肢が spec §3-A どおり | 9カテゴリの選択肢・既定値・v1本数の天井（§3-B）が spec の確定値と一致する |
| 1-5 | 既定値の控えめ投入（§0-3） | `mood_natural`・`tone_muted`・`light_left`・`comp_left` 等の既定が初期状態で入っており、初期プロンプトが成立する |
| 1-6 | 自由入力欄（②被写体・具体物） | 入力文字列がトリムされてそのまま fragment に差し込まれる（加工されない＝決定論） |
| 1-7 | 未選択カテゴリのスキップ（§4-2） | 未選択カテゴリは出力からスキップされ、区切り文字（カンマ/読点）が二重に出ない |
| 1-8 | 語順固定（§4-1） | 画像タイプ→被写体→構図→光→質感→雰囲気→不完全さ の固定順で連結される |

## 2. 被写体タイプ分岐テスト（実機検証の最重要設計 §2-3）

| # | 観点 | 合格条件 |
|---|---|---|
| 2-1 | ②被写体で person/object/scene が確定 | 選んだ被写体タイプが answerState に保持され、以降のカテゴリの出し分けを駆動する |
| 2-2 | 人物（person）= 光主役 | ④光の `light_soft`（柔らかい影）が既定オン、⑤質感は film grain のみ任意、⑦に人物専用項目（毛穴・目の反射・非対称・年齢）が出る |
| 2-3 | 物（object）= 質感主役 | ⑤質感の `tex_grain` が既定オン、④光は方向＋影のコアのみ（硬軟は任意）、⑦は candid/生活感のみ |
| 2-4 | 空間（scene）= 質感主役 | ⑤質感の `tex_grain` が既定オン、④光は弱め、⑦は生活感・環境文脈 |
| 2-5 | 被写体別ネガティブの厚み | 人物時は解剖系（指破綻・bad anatomy・dead eyes/プラ肌除去）が厚く、物/空間は軽め（§3-A ⑧）になる |
| 2-6 | 被写体を切り替えた時の再構成 | 被写体を変更すると出し分け・既定が切り替わり、プレビューが破綻なく再構成される |
| 2-7 | 肌質感の二重出力なし（§4-3） | ⑤`tex_skin` と ⑦`person_pores` が両オンでも、同一/近接の語が de-dup され1つに統合される |

## 3. 出力・受け渡しテスト（受け渡し型の核）

| # | 観点 | 合格条件 |
|---|---|---|
| 3-1 | 3系統独立コピー（§6-3） | 日本語版・英語版・ネガティブの3つが**それぞれ独立したコピーボタン**で正しくクリップボードにコピーできる |
| 3-2 | コピー成功フィードバック | コピー実行時に成功表示（第1弾資産流用）が出る |
| 3-3 | 日英対構造の整合（§1-2） | 日本語版と英語版が**同じ選択内容を指す**（fragmentJa/fragmentEn が対で、言っている内容がズレない） |
| 3-4 | ネガティブは英語のみ・重み記法なし（§5・★3） | ネガティブ欄は英語語句のカンマ区切りで、`:1.3` 等の重み記法が付かない |
| 3-5 | ネガティブ注記の表示（§5） | ネガティブ欄直下に「対応AIのみ有効・MJは`--no`/SDはネガティブ欄」の注記が出る |
| 3-6 | アスペクト比は本文に入らず注記（§8・⑨） | 比率はプロンプト本文に書かれず、「お使いのAIの記法で {選んだ比率} を指定」と値だけ注記に出る |
| 3-7 | **参考AI注記が年付きで出る（★重要・受け渡し型）** | 完成プロンプト直下に「画像生成に強いAIの例（**2026年時点**）: Flux 2 / Imagen 4 / GPT Image 2 など。モデルは入れ替わるため…」が**年付き・例示・非断定**で表示される |
| 3-8 | 参考AI注記の定数集約（§6-3 ★D） | モデル例と「2026年時点」が1箇所の定数/1ファイルに集約され、更新が1点修正で済む構造になっている |

## 4. ライブプレビュー・決定論テスト（信頼性の核）

| # | 観点 | 合格条件 |
|---|---|---|
| 4-1 | ライブプレビュー（§4-4） | 回答するたびにプレビューが再描画され、答えるほど文章が育つ（途中でも常に「今の完成形」が見える） |
| 4-2 | 日英トグル（出力プロンプト言語・§6-2） | トグル1つで日本語版/英語版を切替表示でき、初期値=英語。日本語版も1クリックで全文見える |
| 4-3 | **決定論=同入力同出力（§4-5・最重要）** | 同じ answerState（被写体・全選択・自由入力文字列が同一）なら、リロード後・別ブラウザでも**必ず同じ3文字列**が返る |
| 4-4 | 乱数/API/時刻/LLM 不使用（§4-5） | buildPrompt/buildNegative が乱数・`Date.now()`・外部API・LLM を一切使わない（コードレビューで確認） |
| 4-5 | 入力を送信しない（受け渡し型・プライバシー） | ネットワークタブで、回答・自由入力・生成プロンプトが**外部送信されない**ことを確認 |
| 4-6 | 任意項目の注記表示（§7-2） | 時間帯/天候欄（単独だと過剰・硬軟とセット推奨）・人物肌質感欄（高品質AIは素で十分・気になる時だけ）の注記が各欄に出る |

## 5. アクセシビリティ（WCAG AA）

| # | 観点 | 合格条件 |
|---|---|---|
| 5-1 | キーボード操作 | 質問の選択・前へ/次へ・トグル・コピーがキーボードだけで完結する |
| 5-2 | フォーカス可視 | 全操作要素にフォーカスリング（共通枠 `--color-focus-outline`）が出る |
| 5-3 | コントラスト | テキスト・UI要素が AA 比率を満たす（ライト/ダーク両テーマ） |
| 5-4 | スクリーンリーダ | 選択状態・プレビュー更新・コピー成功・注記が読み上げられる（aria-live でプレビュー/結果を通知） |
| 5-5 | 言語不問UIへの配慮（§6-5・U-05） | 各カテゴリにアイコンが添えられ、選択肢が言語に頼らず識別できる（推奨） |
| 5-6 | フォーム入力 iOS ズーム回避 | 自由入力欄が16px維持でフォーカス時にズームしない（design-concept §2-2） |

## 6. レスポンシブ

| # | 観点 | 合格条件 |
|---|---|---|
| 6-1 | モバイル（〜767px） | ステップ式質問とライブプレビュー（質問の下 or 切替）が縦積みで崩れず操作できる |
| 6-2 | タブレット〜デスクトップ | 768px 以上で質問エリアとプレビューエリア（右側 or 並列）が整合（共通枠768px境界＝design-concept.md） |
| 6-3 | clamp/トークン準拠 | 余白・タイポが clamp＋space スケールに乗り、ビューポート間で破綻しない |
| 6-4 | 長文プレビューの折返し | 英語/日本語の長いプロンプト文がプレビュー欄で折り返し、横スクロール・はみ出しが起きない |

## 7. 表示基盤・規範準拠（static-template / Reid 照合）

| # | 観点 | 合格条件 |
|---|---|---|
| 7-1 | static-template 基盤整合・FOUC無し | `dev/` のテンプレ基盤（SCSS glob import・`_viewport.js`・`<head>` link クリティカルパス）に乗り、初期表示で FOUC（スタイル未適用のちらつき）が起きない |
| 7-2 | **kiso.css import 地雷チェック（★再発防止）** | テンプレ由来 `_reset.scss` 1行目の `@import "kiso.css";` が**削除されている**（残ると本番でCSS 404/MIMEエラー＝メモリ `reference_static_template_kiso_import_gotcha`）。`grep -ri kiso` でヒット0 |
| 7-3 | viewport メタ | `<meta name="viewport">` が設定され、モバイルで等倍表示される（素CSS版でviewport欠落の前例あり） |
| 7-4 | FLOCSS 命名 | クラスが FLOCSS（接頭辞・層）規範に準拠し、共通枠のクラス体系と整合 |
| 7-5 | デザイントークン | 色・余白・タイポが本体派生トークン（tokens.css）を使い独自色を作らない（ツール固有 `--tool-panel-*`/`--field-*` を使用＝design-concept §2-2） |
| 7-6 | 共通枠との一貫性 | ヘッダー/フッター/パンくず/制作者カード/CTA の共通枠（b-tech-cool・768px境界・ダーク追従）に乗っている |
| 7-7 | パンくず表示 | `制作者ポートフォリオ ＞ ツール一覧 ＞ AI画像プロンプト・ビルダー` が表示され、JSON-LD の name と一致（03_seo §5-3） |
| 7-8 | WASM/Worker 非依存（軽量基盤） | 重いWASM依存が無く（決定論ビルダー＝純粋関数）、初回ロードが速い（spec §10） |

## 8. コピー・表現QA（過剰約束の禁止・★ブランド整合）

| # | 観点 | 合格条件 |
|---|---|---|
| 8-1 | **過剰約束の禁止（§0-2・§7-3・最重要）** | UI文言・見出し・H1・リード・注記・CTAに「必ず」「絶対」「AI臭さを消す（断定）」「実写になる」が**一切ない**。「近づける」「抑える」「織り込む」「組み立てる」で統一 |
| 8-2 | 注記の標準セットが揃う（§7-2） | 時間帯/天候・人物肌質感・ネガティブ・アスペクト比・参考AI の5箇所の注記がすべて表示される |
| 8-3 | 固有名詞・誤情報なし | 対外文言にオンラインスクール名等の固有名詞が無い。モデル名（Flux 2 等）は参考例として年付き・非断定で出すのみ（断定的な推奨をしない） |
| 8-4 | 受け渡し型の正直な表現 | 「プロンプトを組み立てるところまで」「生成はユーザーが各自のAIで」が崩れていない（生成・実写化を約束しない） |
| 8-5 | カテゴリ見出しの平易さ（§7-1） | カテゴリ見出しが専門用語に頼らず具体的（「光の方向」「背景をぼかす」等）で初学者にも分かる |
| 8-6 | SEOメタとの整合 | `<title>`/description/OGP が 03_seo §2〜4 の確定文言と一致し、過剰約束が混入していない |

## 1. 実現性（決定論ビルダー単体・ブラウザ完結で overview 機能が無理なく作れるか）

確定方式（API接続なし・ブラウザ完結・無料・キー不要）の範囲内で、overview の主要機能はいずれも**素直に実装できる**。第1弾の山場（複数WASMエンコーダ・Web Worker逐次・AVIF固まり）に相当する**重い依存・重い計算は無い**。

| overview 機能 | 実現手段（決定論ビルダー単体の範囲） | 実現性所見 |
|---|---|---|
| 9カテゴリ質問フロー（ステップ式・1画面1〜数問） | フォーム状態管理＋ステップ遷移（前へ/次へ）。素のJS or 軽量で足りる | ✅ 重い依存なし。第1弾のような WASM/Worker 不要 |
| 各回答→プロンプト文の決定論生成（同入力＝同出力） | **回答キー→プリセット文の対応表＋文字列組立（テンプレートエンジン）**。乱数・API・LLM を一切使わない | ✅ 確定方式の核。再現性は「対応表＋連結」で自然に担保される |
| ライブプレビュー（答えるほど文章が育つ） | 状態変化→プレビュー再描画（純粋関数で state→prompt 文字列を導出） | ✅ 決定論なので副作用なく描ける。技術的落とし穴なし |
| 日英＋ネガティブの独立コピー（3系統） | **同一回答 state から日/英/ネガの3文字列を導出**＋Clipboard API で各欄独立コピー | ✅ 実装容易。第1弾でコピー成功フィードバックは実装済み資産あり |
| 参考AI注記（年付き・例示・非断定） | 静的テキスト（完成プロンプト直下のノート） | ✅ 実装ゼロ難度。**ただし陳腐化＝コンテンツ運用の論点**（★D） |

**技術的な落とし穴 / 重い依存の有無**:

- **重い依存＝無し**。WASM・Web Worker・大容量バイナリ配信は不要（第1弾の `base:"/tools/"` × Worker内WASMパス解決のような地雷が、本ツールには存在しない）。実現性リスクはエンジニアリングではなく**コンテンツ（知識ベース）側**に寄る——overview §6 ★B の自己認識は正しい。
- **唯一の実装上の注意（軽微）**: **日英2言語×ネガティブ＝3系統を、1つの回答 state から決定論的に導く対応表の二重持ち**（日本語プリセット文／英語プリセット文を回答キー単位で対になるよう管理）。ここがズレると「日本語版と英語版で言っている内容が違う」事故になる。これは技術ではなく**データ設計（プリセット辞書の構造）**の問題で、spec の対応表設計で吸収すべき（→ §5 ★B/★C と連動）。実現性そのものは阻害しない。
- **static-template 基盤適合**: overview §6 ★F の「エンジニアリングは軽い／WASM等の重い依存なし／第1弾同様 static-template を F3 で起こす」は妥当。第1弾でテンプレ基盤フルリビルド（FOUC・viewport 根因解消）が済んでおり、本ツールはその上に**フォーム＋テンプレートエンジン＋コピー**を載せるだけ＝**基盤リスクは第1弾で既に踏破済み**。

> 実現性の指摘はすべて「決定論ビルダー単体・ブラウザ完結」の範囲内で行った。API前提の評価はしていない（API接続は §2-10 で「やらない」確定）。

## 2. 工数 / スコープ（完成形前提での工数感・過大/過小・二層構造の妥当性）

> 工数スケール（第1弾 §3 と同基準）: S=〜3日 / M=1〜2週 / L=2週超。

### 2-1. 第1弾比の工数感（前提＝完成形スコープ／**知識ベース執筆を含むか否かで二分**）

**前提を明記する。本ツールの工数は「実装」と「知識ベース執筆（craft言語化＋裏取り＋実機検証）」で性質が全く異なる**ため、分けて見る。

| 区分 | 工数感（ラフ） | 第1弾比 | 根拠 |
|---|---|---|---|
| **A. 実装（Cody / F3）** | **S〜M下振れ（実働 3〜6日目安）** | **第1弾より軽い** | フォーム＋ステップ遷移＋state→prompt の純粋関数＋3系統コピー＋ライブプレビュー。WASM/Worker/AVIF固まり対策が無い分、第1弾（実働2〜3週）より明確に軽い。基盤は第1弾で踏破済み |
| **B. 知識ベース執筆（craft / 上流）** | **M〜L（実働 1.5〜3週目安・実機検証込み）** | **第1弾の「種別⇄パラメータ表」より重い** | AI臭さ消し対応表（光・質感・構図・不完全さ・人物リアリティ⇄日英プリセット文）の一次案＋裏取り＋**実生成での当て確認**。9カテゴリ×日英×ネガの文面を「実際に効く」水準で書き切る作業。ここが本ツールの主役（§3 で詳述） |

**総工数感（完成形・知識ベース込み）**: **M上振れ〜L（実働 2.5〜4週目安）**。内訳の重心が第1弾と逆転している点が最大の特徴——**第1弾は実装が山場、第2弾はコンテンツ（craft）が山場**。「エンジニアリングは軽い／上流が肝」という overview の自己認識（§2-10 実装難所・overview §6 ★B）は工数構造として正しい。

### 2-2. 完成形スコープが過大/過小になっていないか

- **過大ではない**: 9カテゴリは「AI画像生成プロンプトの構成要素」として網羅的だが水増しではない（タイプ／被写体／雰囲気／光／質感／構図／不完全さ／ネガティブ／アスペクト比＝画像プロンプトの標準的な軸）。完成形＝ミニマルにしない（§2-10）の指示に対し、過剰な機能を足してもいない（編集・保存・履歴・アカウント・LLM対話は正しく「やらない」）。
- **過小でもない**: コア5問だけで止めず、AI臭さ消しの核（段4〜7）を完成形に含めている＝「完成形で出す」の趣旨に合致。
- **唯一の過大化リスク＝知識ベースの深さ**: カテゴリ数（横幅）は妥当だが、各カテゴリのプリセット文の**選択肢数（縦の深さ）を無制限に広げると B の工数が青天井**になる。これは spec で「v1で各段に用意するプリセット文の本数」を有限に切ること（条件②）。横幅9カテゴリは固定でよいが、縦の深さは v1 で天井を引く。

### 2-3. 「コア数問＋任意深掘り」二層構造は工数管理上妥当か

**妥当。むしろ工数管理上の安全弁として優秀**。第1弾で Mane が評価した「任意機能への降格ライン（劣化ルーペ＝工数を見て載せる／自動推測＝最後に積む）」と同じ思想:

- **コア5問（タイプ・被写体・雰囲気・光・構図）で v1 が成立する**設計＝深掘り4段（質感・不完全さ・ネガティブ詳細・出力指定の作り込み）が工数オーバー時に**後回し/簡素化できる切り離し線**になる。
- 実装順序として「コア5問のフロー＋プレビュー＋3系統コピー」を先に通せば、深掘り段の知識ベースが間に合わなくても v1 は出せる。**工数オーバー時の安全弁が構造的に効いている**。
- 提言: spec で **「コア5問＝v1必達／深掘り4段＝知識ベースの仕上がりに応じて段階投入」** と明記し、第1弾の「実装は最後に積む」と同じ降格ラインを knowledge base にも引くこと。

**工数を超えて膨らんでいる箇所**: 実装側＝なし。**過小評価の箇所**: 知識ベース工数（B）が overview では「主役」と認識されているが**工数レンジが未提示**＝spec 着手前に体制と工数を確定しないと、ここだけが超過の温床になる（→ §3）。

## 3. 本質リスク＝知識ベース（craft）の工数・担い手・進め方

overview §6 ★B が「知識ベースは本ツールの主役／第1弾の机上仕様→実機書き換え教訓を踏まえ、プリセット文は実生成で当て確認が安全」と自己認識している点は**正確**。ここが本評価で最も強く言いたい論点。

### 3-1. 工数の見立て

- **B（知識ベース）= M〜L（実働 1.5〜3週・実機検証込み）**。理由: 9カテゴリ×日英2言語×（プリセット文＋ネガティブ）の文面を、**「AI画像生成に実際に効く」水準**で書く必要があり、机上で書いた文面が実機で効かない確率が高い（第1弾の教訓と同型）。「光の方向指定」「フィルム粒状感」「三分割・余白」等が**どのモデルでも一定効く汎用表現**になっているかは、実生成で当てないと判断できない。
- これは「文章を書く」工数ではなく「**書く→実生成→効かない→書き直す**」の反復工数。第1弾の「リサイズ上限px・品質値を実画像で調整」と同じ構造を、コンテンツ側で踏む。

### 3-2. 誰が担うか（Sara / Mado / 黒田さん）

役割が混在しているので spec 着手前に切り分けること（★B の中身）:

| 作業 | 担い手（提言） | 理由 |
|---|---|---|
| **craft の一次案（AI臭さ消しの知見→対応表の骨子）** | **黒田さん＋Mado** | 「AI生成画像をWebに何度も置いて違和感を踏んだ実務知」（overview §5）は黒田さんの一次情報。Mado が構造化（カテゴリ⇄プリセット対応表に落とす）。**Cody/Sara では一次案は書けない**（実務知の出所が黒田さん） |
| **日英プリセット文の文面整形・コピー品質・過剰約束チェック** | **Sara** | 対外公開文言の品質（事実ベース・過剰約束なし＝§2-10 コピー方針）はコンテンツ部の領分。英語プリセットの自然さも含む |
| **実機での当て確認（実際に画像生成して効くか）** | **黒田さん（最終）＋ 一次は Mado** | 実生成環境（各自のAI）を持つのは黒田さん。**ここを省くと第1弾教訓の再発**。spec の段階で「どのモデルで何ケース当てるか」のミニ検証計画を決める |

> **★要確認（黒田さん）**: 知識ベース一次案の主担当を「黒田さん発の知見を Mado が構造化」で進めてよいか。実機検証（実生成で当てる）に黒田さんの工数（数時間〜1日規模）を**spec 段階で確保できるか**。ここが確保できないと B が机上仕様のまま固まり、公開後に「効かないプロンプト」を出すリスク（ブランド毀損）。

### 3-3. 第1弾教訓を踏まえた進め方の提言

第1弾の「机上仕様→実機で書き換え」（CF7記事でも同型の教訓あり）を**第2弾で繰り返さない**ための具体手順:

1. **spec 段階で「実機検証を knowledge base の品質ゲートにする」と明文化**（机上で全プリセットを書き切ってから実機、ではなく**書く→即当てる**の小ループ）。
2. **薄い縦スライスで先に当てる**: 9カテゴリ全部を書く前に、**1カテゴリ（例＝光）だけ日英プリセットを書き→実生成→効くか確認**し、対応表の「型」を検証してから横展開する（第1弾の「最初に AVIF 固まりを検証」と同じ＝最大リスクを先に潰す）。
3. **「効かなかった文面」をログ化**して spec/knowledge base に残す（ネガティブ知見も資産。賞味期限管理 ★D と接続）。
4. **F3 実装と knowledge base を分離発注**: 実装（A・Cody）は対応表が**プレースホルダ（仮プリセット）でも着手可能**な構造にし、knowledge base（B）の確定を待たずに UI/フロー/コピー機構を先に通す。これで A と B を並行でき、B の反復が A をブロックしない。

## 4. マネタイズ整合（§2-9 広告フリー・リード本命と矛盾しないか）

**矛盾なし。** overview は本ツールUIを「広告フリー・無料・キー不要・入力を送信しない」受け渡し型として設計しており、§2-9 の「ツール本体＝完全に広告フリー／役割＝リード獲得・スキル証明・ドメイン育成・SEO destination」に正対する。

- ツールUI自体に広告・寄付・課金導線を一切置いていない（§2-9 の「広告動画・端AdSense・寄付リンクは載せない」と一致）。
- 直接収益＝frontend-note 記事側のAdSense（§2-9）＝overview スコープ外で正しく触れていない（混入していない＝良い）。
- **むしろリード本命との整合は強い**: 「フリーランスWeb屋の知見を質問設計・知識ベースに落とす」差別化（overview §5）＝**スキル証明そのもの**で、§2-9 の「若い営業の顔を育てる／スキル証明への投資」と本質的に合致。広告フリーで知見を惜しみなく出すほどリード価値が上がる構造。
- 提言（軽微）: §2-9 の「記事⇄ツールの相互送客リンク」に乗るため、**本ツール用の frontend-note 記事（「AI画像のAI臭さを消すプロンプトの作り方／本ツールの使い方」）を 03_seo（Sara）で射程に入れる**こと（★G と接続）。これはマネタイズ動線（記事AdSense＋リード送客）の器になる。

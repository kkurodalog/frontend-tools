## 重点観点の評価

### 観点1: works非混在の担保（PLAN §2-4）— ◎ 合格

- tools は**独立した `<section class="l-section" id="tools" aria-labelledby="tools-title">`**（index.html L111）。works セクション（`#works` L30-108）の `c-grid--triad`（L37-104）の中には1枚も混入していない。works の `</section>`（L108）で完全に閉じた後に tools セクションが開く構造で、PLAN §2-4「works 混在は不採用＝混ぜると営業文脈が濁る」を構造レベルで満たす。
- tools カードのリンク先は `/tools/image-compress/`（L121）・一覧 CTA は `/tools/`（L142）で、works 側（`/works/...`・`/works/`）と完全に分離。導線が混ざる箇所もない。**満点**。

### 観点2: works対等並列（実績と並ぶもう一つの成果＝技術力の傍証）— ◎ 合格

- tools は works 直後（works L108 閉じ → tools L111 開始）の独立 `<section>` として配置。works に**ぶら下げず**、同じ骨格（`c-section__head` > `c-section__title`＋`c-section__lead` / `c-grid--triad` / `c-work-card` / `p-center-link`>`c-link-arrow`）で組まれており、視覚的にも構造的にも works と**対等な兄弟セクション**になっている。PLAN §2-4 #2「実績の下にぶら下げず対等並列」を満たす。
- 配置順「works（請けた仕事）→ tools（自分で作った道具）」が隣接することで「実績と並ぶもう一つの成果＝技術力の傍証」という情報設計が成立。さらに「works → tools → about → services」という流れで、ツールが about（人物）の前＝「成果物の証拠群」のブロック内に収まり、信頼形成のリズムも自然。**満点**。

### 観点4: コピーの訴求・トーン — ◎ 合格

- **見出し「公開ツール」**（L114）: works の「最新の制作実績」と対をなす素直な名詞句。誇張なし・�leadへの橋渡しとして機能。
- **リード文「フロント制作の実作業で使う自作ツールを公開しています。」**（L115）: 黒田さん確定文言と一字一句一致。positioning（§2-8 フロント制作者向け）を「フロント制作の実作業で使う」で明示し、「自作」で技術力の傍証＝営業資産（§2-9）を匂わせつつ、売り込み語（「便利」「最強」「無料」等）を使わない誠実トーン。ブランドガイドラインの「売り込みすぎない」と整合。
- **カードの tagline「画像の役割ごとに最適な圧縮を当てる。ブラウザ内で完結＝画像を送信しない。」**（L134）: 機能の核（役割別最適化）とプライバシー優位（送信しない）を端的に提示。誇張・断定リスクなし。
- **badge「画像圧縮」「WebP」**（L136-137）: 機能カテゴリの事実記述。works カードの badge（「WordPress」「オリジナルテーマ」等）と内容が明確に異なり、tags の内容差で works/tools が自然に弁別される。
- **誤認リスク**: 現状 published 1件を「ツール（複数）」と読ませる過大表現は無し。見出し「公開ツール」は集合名詞だが、カードが1枚かつ「ツール一覧を見る →」が単数複数を断定しないため、1件を複数と誤認させる炎上リスクはない。**満点**。

### 観点5: works/tools カードの視覚弁別（Should3）— ○ 合格（運用注記は Should1 へ）

- Cody の判断「見出し＋背景分離＋tags内容差で十分・弁別バッジ不要」は、**現状 N=1 の局面では情報設計上妥当**。根拠:
  - セクション見出し「公開ツール」＋リード（L113-116）で文脈が明示的に立つ。
  - 背景交互（works=alt / tools=base）で面が分離（観点3）。
  - tags が「画像圧縮/WebP」vs「WordPress/オリジナルテーマ」で内容弁別。
- 前回評価 Should3 で私が挙げた「スクロール中の瞬間視認で区別しにくい」副作用は、**works=alt 直後に tools=base が来る背景切替**で実質的に緩和されており、バッジ追加は現状では冗長というCody判断に同意する。
- ただしこれは「カードが各セクション1〜数枚で、見出し・背景という強い文脈が常に近接している」前提で成立する。将来 tools/works が両方とも triad 3枚に育つと、スクロール中にカードだけが視界に入る局面が増え、同一 `c-work-card` 見た目の弁別が弱くなる（→ Should1 に運用注記として残す）。

### 観点6: ファネル整合（認知→信頼→行動）— ◎ 合格

- 導線: カード → `/tools/image-compress/`（個別ツール）／一覧 CTA `p-center-link`>`c-link-arrow`「ツール一覧を見る →」→ `/tools/`（ポータル）。
- ファネル: トップで「公開ツール」を**認知** → works と対等並列＝技術力の傍証として**信頼**形成 → 個別ツール or 一覧へ**行動**、という認知→信頼→行動の流れに沿う。本体（ポートフォリオ）に来た訪問者を `/tools/` へ送る順方向営業導線として機能し、PLAN §2-9「ツール=営業資産・スキル証明への投資」と整合。
- リンク先 `/tools/`・`/tools/image-compress/` は frontend-tools 側で実在（前回評価で裏取り済み）。導線が宙に浮かない。**満点**。

### 観点7: 抜け漏れ・過剰 — ○ 合格

- **F5② として出すべき要素**: 見出し・リード・カード（N=1）・一覧 CTA・サムネ画像（webp+fallback）・h2 兄弟チェーン（aria-labelledby）・背景交互、すべて揃っている。不足なし。
- **過剰**: 新規 SCSS（`_p-tools-preview.scss`）を作らず既存クラス流用で済ませた判断は §6② 最小方針どおり。過剰な装飾・新規トークンの持ち込みなし。
- **h2 兄弟チェーン**: works-title → tools-title → about-title → services-title → cta-title のフラット兄弟（L33/L114/L150/L176）。`aria-labelledby="tools-title"` ↔ `<h2 id="tools-title">`（L111/L114）対応も正しく、見出しアウトラインが崩れていない（前回 Should4 充足）。

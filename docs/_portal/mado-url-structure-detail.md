# ツールURL構造 詳細判断 — サブドメイン vs サブディレクトリ / works整合 / DB時のサブドメイン要否

前提資料: `mado-publish-format.md`（案B=ツールポータル型 確定 / §2-1 サブドメインvsサブディレクトリSEO原則 / §5-2 URL設計案）

## 0. 結論（先出し）

- **問1**: SEO・運用ではサブディレクトリが有利（評価集約・ゼロスタート無し）。サブドメインの強みは「性格分離」と「独立運用」。本件は**両者が拮抗**するが、ドメインパワーが若い今は集約メリットが効きにくく、判断は「性格分離をどれだけ重視するか」に収束する。
- **問2**: works と**揃える必要はない**。worksは「ポートフォリオの構成要素＝本体に在って当然」。ツール集は「別の検索意図・別の被リンク文脈を持つdestination」。性質が違うので構造一致自体に価値はない。ただし結論として**ツール集も本体サブディレクトリ `kurodafolio.com/tools/` に置くのが最適**（理由は問1総合）——これは「worksに揃えたから」ではなく「ツール集の性質を独立に評価した結果」たまたま同じ構造になるだけ。
- **問3**: 黒田さんの認識「DBが要るならツール単位でサブドメイン」は**部分的にしか正しくない／多くのケースで不要**。正しい切り分けは「**ユーザーに見えるUIページ（SEO対象）」と「バックエンド/API/DB（SEO不要・インフラ）」を分離**すること。UIはポータル配下に残し、DB/APIは別オリジン（`api.kurodafolio.com` やCloudflare Workers等）にfetchで逃がせばよい。**DBが要ってもUIはサブドメインを切らずポータルに残せる**。サブドメインを丸ごと切るべきは「認証・アカウント・SSRを伴う重い動的アプリ全体」など限定的ケースのみ。
- **最終推奨**: `kurodafolio.com/tools/`（本体サブディレクトリ）。第1弾「画像仕分け圧縮くん」はWASMブラウザ内完結＝DB不要で問3の影響を受けない。

※ 注: `mado-publish-format.md §5-2` では Mado推奨を「性格分離を取って `tools.`」としていたが、本詳細検討で「本体ドメインパワーが若い／works等の本体コンテンツが既にサブディレクトリで同居している」事実を精査した結果、**推奨を `kurodafolio.com/tools/` に更新**する（理由は問1〜問2参照）。前回の★要確認を、本書で詰めて結論化した形。

## 問1: サブドメイン vs サブディレクトリ 詳細比較

### 1-1. 観点別メリデメ表

| 観点 | `tools.kurodafolio.com`（サブドメイン） | `kurodafolio.com/tools/`（サブディレクトリ） |
|---|---|---|
| **SEO: ドメイン評価の集約/分散** | 別サイト扱いされやすく、評価がサブドメイン内に留まる傾向。本体の評価を完全には継承しない | 本体ドメインの評価を**そのまま継承**。ツールが得た被リンク・評価も本体に還元され**1ドメインに集約**（◎） |
| **SEO: 立ち上がり速度** | 実質ゼロ〜準ゼロスタート。本体評価の恩恵が薄く、自力で評価を積み直す期間が要る | 本体の既存評価（works公開以降の被リンク・インデックス）を**初日から相続**。立ち上がりが速い（◎） |
| **SEO: 被リンク継承** | ツールが集めた被リンクは主にサブドメインに効く。本体↔ツール間の評価波及が弱い | ツールの被リンクが本体パワーを押し上げ、本体の被リンクもツールに効く**相互波及**（◎） |
| **ブランド: 性格分離 vs 統合** | 「営業の顔（ポートフォリオ）」と「便利ツール集（destination）」を**性格分離**できる（◎）。本体URLを汚さず独立ブランドとして育てやすい | 本体ドメイン直下にツールが同居。本体の性格が「ポートフォリオ＋ツール」に**やや拡散**。ただしworksも記事も同居している現状、既に「複合サイト」化しているため希釈度は限定的 |
| **技術: SSL** | XServer無料SSLはワイルドカード/サブドメイン個別とも対応。サブドメイン分の証明書手当が要る（自動化されていれば軽微） | 本体証明書1枚でカバー。**追加SSL手当ゼロ**（◎） |
| **技術: GA4分離容易性** | サブドメインは別propertyにも同一property（クロスドメイン/サブドメイン計測）にもできる。**ツール集だけの計測を綺麗に切り分けやすい**（◎） | 同一property内でパス（`/tools/`）フィルタで分離。設定すれば分離可能だが、サブドメインより1手間。GA4自体はサブドメイン跨ぎを標準で1propertyに統合計測できる |
| **技術: Search Console管理単位** | ドメインプロパティ（`kurodafolio.com`）配下に自動で含まれる。URLプレフィックスで `tools.` を別登録すれば**ツール集だけのCWV/カバレッジ/クエリを独立監視**できる（◎） | 同じくドメインプロパティに含まれる。`/tools/` だけの独立監視は、URLプレフィックスプロパティを別途切れば可能だがパス単位の切り出しはサブドメインより粒度が粗い |
| **技術: デプロイ独立性** | DNS・ドキュメントルートを分ければ**本体と完全独立にデプロイ**可。ツール側の事故が本体に波及しない（◎） | 本体リポジトリ/ビルドに同居 or サブパスを別ビルドで配置。独立は可能だが本体デプロイと結合しやすい。設計次第 |
| **運用: 保守** | サブドメイン1個分のDNS/SSL/解析設定の初期コスト。以後はポータル内でサブディレクトリ集約するので増殖はしない | 追加サブドメイン設定ゼロ。本体運用にそのまま乗る（◎） |
| **将来の引っ越しコスト** | 後から本体配下に移すと301・被リンク毀損が発生 | 後からサブドメインに切り出すのも301発生。どちらも「最初に決めて動かさない」のが鉄則 |

### 1-2. 「ドメインパワーが若い」ことがこの選択に与える影響

**結論: 本件では「本体継承の旨味が現状小さい」のではなく「だからこそ集約して育てるべき」side に効く。**

- 一般論として「本体がまだ弱いならサブディレクトリ継承の旨味は小さい（継承元が貧弱）」という見方はある。だがそれは**静的な見方**。重要なのは**今後どちらが速く育つか**という動的な見方。
- 本体は2026-05-31公開と若く、works公開・今後のブログ連携で**これから評価を積む局面**。ここでツールを別サブドメインに切ると、**本体が積む評価とツールが積む評価が2拠点に割れる**。若いドメインを2つ並行で育てるのは、個人事業の被リンク獲得力では非効率（弱い苗を2本に分けて育てるより1本に集約した方が太る）。
- 逆にサブディレクトリで同居させれば、**ツールが集めた被リンク・流入が若い本体ドメインの底上げに直結**し、本体（ポートフォリオ＝営業の顔）の検索評価も一緒に育つ。営業の顔を育てたい黒田さんの事業目標とSEOの方向が一致する。
- すなわち「若いドメインだからこそ評価を1拠点に集約し、相互に育て合うサブディレクトリが有利」。**ドメインパワーが若い事実は、サブディレクトリ側を後押しする。**

### 1-3. 問1の小結

SEO（集約・継承・立ち上がり）・SSL・運用の各面で**サブディレクトリが優位**。サブドメインの優位は「性格分離」「解析/SC独立監視のしやすさ」「デプロイ完全分離」の運用上の綺麗さに限られる。本体が若い今は**集約メリット＞性格分離メリット**と評価する。

## 問2: works（サブディレクトリ）と揃えるべきか

### 2-1. 性質の違いを言語化する

| | works（制作実績） | tools（便利ツール集） |
|---|---|---|
| **役割** | ポートフォリオの**構成要素**。「この人は何を作れるか」を示す営業資料そのもの | 制作物ではなく**提供サービス/destination**。「ここに来れば便利な道具がある」拠点 |
| **本体との関係** | 本体（営業の顔）に**在って当然**。worksが無いポートフォリオは成立しない。本体と不可分 | 本体に在らねばならない必然はない。本体（ポートフォリオ）とは独立に成立しうる別コンテンツ |
| **検索意図** | 「黒田 制作実績」「kurodafolio 事例」等、**指名・営業文脈**。本体ブランドを知っている人が辿る | 「画像圧縮 ツール」「clamp 計算」等、**汎用・課題解決文脈**。黒田さんを知らない不特定多数が検索で着地 |
| **被リンク文脈** | 案件・人物・ポートフォリオ文脈で貼られる | 「便利ツール」「制作お役立ち」文脈で貼られる（本体とは別系統の被リンク） |
| **更新頻度/性格** | 案件完了ごとに静的に追加。営業資料として安定 | ツール追加・機能改善で能動的に育てるプロダクト的性格 |

### 2-2. 「構造を揃えること」自体に意味はあるか

- **ユーザー体験**: worksとtoolsは利用文脈が別（営業文脈で実績を見る人／課題解決でツールを使う人）。両者を**同じ階層に並べる必然はない**。URL構造の一致がユーザーの理解や回遊を助ける場面は実質ない。
- **SEO**: 「揃っているから評価が上がる」という効果は無い。SEOに効くのは構造の一致ではなく**評価の集約（1ドメインに束ねるか）**。これは問1の論点であって「worksと揃えるか」とは別問題。
- **運用**: worksもtoolsも同じstatic-template系（Vite+Handlebars+FLOCSS）で組むなら、**同一ドメイン配下に置く方がリポジトリ・ビルド・デプロイ・SSLが1系統で済み運用が軽い**。これは「worksに揃える」のではなく「同一技術基盤を1拠点に集約する」運用合理性。

### 2-3. 問2の結論（性質ベース・忖度なし）

- **「worksに揃えるべき」という論理は不採用**。worksが本体サブディレクトリにあるのは「ポートフォリオの構成要素だから」であって、ツール集がそれに合わせる理由にはならない。構造一致自体に価値はない。
- **だがツール集を独立に性質評価した結果も、`kurodafolio.com/tools/`（本体サブディレクトリ）が最適**になる。理由は問1（若い本体ドメインへ評価を集約して相互に育てる／SSL・運用が軽い）。
- つまり**「worksに揃えた」のではなく「ツール集の性質を独立に評価したら、たまたまworksと同じ構造（本体サブディレクトリ）が最適だった」**。結論の構造は同じでも、判断の根拠は別。忖度ではなく性質ベースの独立判断としてこの結論に至る。

## 問3: DB/バックエンドが必要なツールはサブドメインを切るべきか

### 3-1. 黒田さんの認識の検証 → 「多くのケースで不要。切り分けが粗い」

黒田さんの認識:「個別DBが必要な場合などは、サブディレクトリではなくツール単位でサブドメインを切った方がいい」。

**判定: 部分的にしか正しくない。** 理由は、この認識が「DBの有無」と「URL（サブドメイン/サブディレクトリ）」を直結させているが、本来この2つは**直交する別レイヤー**だから。正しい軸は「**SEO対象のUIページ**」と「**SEO不要のバックエンド/DB**」の分離。

### 3-2. 核心の切り分け: UIページ（indexable）とバックエンド/API/DB（インフラ）は分離して考える

ツールは技術的に2つの層に分解できる:

| 層 | 中身 | SEO | 適切な配置 |
|---|---|---|---|
| **UIページ層** | ユーザーが検索で着地し操作する画面（HTML/CSS/JS）。タイトル・meta・h1でキーワードを取りに行く**indexable**な資産 | **SEO対象**。流入の入口。**ドメイン評価を集約したい** | `kurodafolio.com/tools/xxx/`（ポータル配下・本体サブディレクトリ） |
| **バックエンド/API/DB層** | データ保存・サーバー処理・認証・重い計算。**ユーザーには直接見えず、検索にもインデックスされない**（むしろインデックスさせない） | **SEO不要**。インフラ | 別オリジン: `api.kurodafolio.com` / Cloudflare Workers / Supabase / 外部API等に置き、UIから**fetchで叩く** |

- **重要な含意**: UIページは「フォーム・ボタン・結果表示の静的HTML＋JS」で、DBアクセスはJSが**裏でAPIにfetchするだけ**。つまり**DBが必要でも、ユーザーに見えるUIページ自体はサブディレクトリ（ポータル配下）に置いたまま**でよい。DBはUIとは別オリジンのバックエンドに逃がせる。
- 具体例: `kurodafolio.com/tools/keyword-saver/`（UIはポータル配下・SEO集約）が、保存処理だけ `api.kurodafolio.com/save` や Cloudflare Workers + D1/KV にfetchする。**UIのURLはサブドメインを切らずポータルに残せる。** 黒田さんの「DBが要る＝ツールごとサブドメイン」という想定は、この分離を知れば**不要になる**。
- なぜこれが正しいか: SEOが評価するのは**UIページ（人が着地するHTML）**であって、バックエンドAPIは元々noindex（検索対象外）。だから「DBの都合」でUIのURLを動かす理由がない。バックエンドは好きなオリジンに置けばよく、UIのSEO設計（ポータル集約）と独立に決められる。

### 3-3. では「ツール単位でサブドメインを丸ごと切るべきケース」は本当に存在するか（デシジョンルール）

存在はするが**限定的**。以下のいずれかに該当する時のみ、ツール全体を独立サブドメイン（例 `app.kurodafolio.com` / `xxx.kurodafolio.com`）に切ることを検討する:

| # | 条件 | なぜサブドメインが妥当か |
|---|---|---|
| 1 | **UIページ自体がSSR/動的生成される重い動的アプリ**（静的HTML＋API fetchで表現できず、サーバーがHTMLを都度生成する） | 静的ポータル（Vite/Handlebars/XServer）の配下に同居させにくい。実行環境（Node/エッジ等）が本体と別系統になり、デプロイ・スケール・障害分離の都合でオリジンを分けた方が安全 |
| 2 | **ログイン/アカウント/個人データを伴うWebアプリ全体**（認証セッション・会員ダッシュボード等） | Cookie/セッションのスコープ、セキュリティ境界（CSP・認証ドメイン）を本体ポートフォリオと分離したい。営業の顔と認証アプリは性格・リスクが別 | 
| 3 | **本体と全く別のホスティング/ランタイムに乗せる必要がある**（例: Vercel/専用サーバーでしか動かないアプリ） | XServer静的ホスティングの配下に物理的に同居できない。オリジン分離が技術的必然 |
| 4 | **本体の可用性とツールの可用性を完全に切り離したい**（ツールが落ちても本体営業ページは無傷にしたい、という運用要件が強い場合） | 障害分離。ただしこれは「やりたければ」の運用判断で、必須ではない |

**逆に、以下なら絶対にサブドメインは不要（ポータル配下サブディレクトリで足りる）:**
- ブラウザ内で完結（WASM/JS処理・DBなし） → 第1弾「画像仕分け圧縮くん」がこれ
- 軽いデータ保存程度（UIは静的、保存だけAPI/Workers/Supabaseにfetch） → §3-2の分離で対応
- 認証なし・SSRなし・本体と同じ静的ホスティングで動く

### 3-4. 第1弾「画像仕分け圧縮くん」への影響確認

- 画像仕分け圧縮くんは**ブラウザ内WASM完結・サーバー処理なし・DBなし**。画像はユーザーのブラウザ内で圧縮され、サーバーに送らない（プライバシー面でも強み）。
- → **問3の論点（DB/バックエンド/サブドメイン要否）の影響をまったく受けない**。`kurodafolio.com/tools/image-compress/` に静的UIとして置けば完結。バックエンドもサブドメインも不要。第1弾は最もシンプルなケースに該当。

## 結論: 最終URL構造方針

### 4-1. Mado推奨（問1〜3総合）

**ツール群は `kurodafolio.com/tools/`（本体サブディレクトリ）に置く。各ツールはその配下のサブディレクトリ。**

```
kurodafolio.com/                       … ポートフォリオ本体トップ（営業の顔）
kurodafolio.com/works/                 … 制作実績（既存・据え置き）
kurodafolio.com/tools/                 … ツールポータル トップ（ツール一覧）  ← 新規
kurodafolio.com/tools/image-compress/  … 第1弾「画像仕分け圧縮くん」（WASM完結・DB不要）
kurodafolio.com/tools/clamp/           … clampメーカー（候補）
kurodafolio.com/tools/color/           … 配色＆コントラスト（候補）
kurodafolio.com/tools/flocss/          … FLOCSSスキャフォルダ（候補）
kurodafolio.com/tools/motion/          … スクロール演出（旧 motion.kurodafolio.com 案を吸収）
```

根拠（一貫した論理）:
- **問1**: 本体ドメインが若い今、評価を2拠点に割らず1拠点（本体）に集約して相互に育てる方が、若い苗を太らせる定石。SSL・運用も本体に乗って軽い。サブドメインの強み（性格分離）より集約メリットが勝る。
- **問2**: ツール集を性質ベースで独立評価しても、本体サブディレクトリが最適（評価集約・運用合理性）。worksと同じ構造になるのは結果論であって、忖度ではない。
- **問3**: DB/バックエンドが要っても、UI（SEO対象）はポータル配下に残し、DBは別オリジンに逃がせる。だから「将来DBツールが出るからサブドメインで統一しておく」という先回りは不要。サブディレクトリで始めて問題ない。

### 4-2. DB/バックエンドを要するツールが将来出た場合の配置ルール（明文化）

**原則: UIとバックエンドを分離し、UIはポータルに残す。**

1. **UIページは常に `kurodafolio.com/tools/{name}/`（ポータル配下）に置く** — indexableなSEO資産として本体に評価を集約する。これが既定。
2. **DB・サーバー処理・認証・重い計算は別オリジンに逃がす** — `api.kurodafolio.com`（XServerで別ドキュメントルート）/ Cloudflare Workers + D1/KV / Supabase / 外部API等。UIからfetchで叩く。**UIのURLは動かさない。**
3. **例外的にツール全体を独立サブドメイン（`xxx.kurodafolio.com`）に切るのは、§3-3の条件1〜4のいずれかに該当する時のみ** — すなわち「UI自体がSSR/動的生成される重い動的アプリ」「ログイン/アカウントを伴うWebアプリ全体」「本体と別ホスティング/ランタイムが必須」「本体と可用性を完全分離したい強い運用要件」。これらに該当しない限りサブドメインは切らない。
4. **迷ったらサブディレクトリ＋別オリジンAPI** — 静的UI＋API fetchで表現できるか最初に検討する。表現できるならサブドメインは不要。

### 4-3. 第1弾の確定

`kurodafolio.com/tools/image-compress/`。WASMブラウザ内完結でDB・バックエンド・サブドメイン一切不要。ポータルトップ `kurodafolio.com/tools/` を「黒田さんのフロントツール集」の顔として同時に用意し、最初の1ツールとして公開する。

### 4-4. 前回検討（mado-publish-format.md）からの更新点（明示）

- §5-2では Mado推奨を「性格分離を取って `tools.`（サブドメイン）」とし、`kurodafolio.com/tools/` を代替案として★要確認に回していた。
- 本書では、本体ドメインパワーが若い事実・works等が既に本体サブディレクトリで同居している現状・SSL/運用の軽さを精査し、**推奨を `kurodafolio.com/tools/`（本体サブディレクトリ）に更新**。「性格分離」より「若いドメインへの評価集約と相互育成」を優先した。
- 案B（ツールポータル型・1ドメイン集約・各ツール=サブディレクトリ）という骨格は不変。変わったのは「集約先を独立サブドメインにするか本体サブディレクトリにするか」の1点で、本書は後者を選んだ。

★要確認（黒田さん最終判断）: 「営業の顔である本体ドメインに便利ツール集を同居させること」への抵抗感の有無。もし「ポートフォリオ（営業）とツール集はブランドを明確に分けたい」という意向が強ければ `tools.` サブドメインも合理的選択肢として残る（性格分離 vs 評価集約のトレードオフ）。SEO・運用の損得勘定だけならMado推奨は `kurodafolio.com/tools/`。

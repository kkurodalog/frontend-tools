## 前提の整理（このカタログの判断軸）

黒田さんが壁打ちで示した嗜好を、案出しの優先順位として次のように解釈した。

1. **第1弾の最優先=「実用性・話題性」**。実力証明・SEO流入は「あれば嬉しい副産物」であって第1動機にしない。「使われる・シェアされる・被リンクされる」が勝ち筋。
2. **第1弾ターゲット=2層**:
   - ①フロント学習者・同業（ブログ読者と同層 / 駆け出し）
   - ②非エンジニア制作関係者（デザイナー・ブロガー・個人事業主 = コードは書かないがWeb素材・軽作業を要する層）
3. **将来はキャリア外も**（投資 / ゲーム・エンタメ / 健康・生活実用）。

### 「需要の証明」の読み方（重要）

検索すると、JSONフォーマッタ・画像圧縮・CSSジェネレータ・OGP画像・文字数カウント・cubic-bezier 等は**すでに無数の競合が存在する**。
これは「やめどき」ではなく「**需要が証明済み**」のサイン。レッドオーシャンほど検索需要は太い。
ただしコモディティ領域で勝つには**明確な差別化（速さ・UX・特化・複数機能の束ね・記事との連動）が必須**。
逆に競合が薄い領域は、需要が細いか・技術的に難しいかのどちらかなので、慎重に見極める。

各案の「差別化ポイント」はこの前提で書いている。

# カテゴリA: キャリア直結・実務ツール（第1弾候補）

> フロント学習者・同業 ＋ 非エンジニア制作関係者向け。実用性・話題性重視。

## A-1. ぱっと画像最適化くん（仮）— Web用画像 一括圧縮＆WebP変換

- **何をするツールか**: 画像をドラッグ&ドロップすると、ブラウザ内で圧縮＋WebP/AVIF変換し、Before/Afterの容量差とプレビューを並べて表示。複数枚一括＆ZIPダウンロード対応。
- **想定ユーザー**: ①駆け出しフロント（表示速度・Core Web Vitals対策で画像軽量化したい）＋②ブロガー・デザイナー（記事用画像を軽くしたいがSquooshのUIは難しいと感じる層）。
- **需要根拠**: 「画像圧縮 WebP変換 無料」は競合がTinyPNG・Squoosh含め非常に多く、まとめ記事も大量＝需要が太い証明。TinyPNGは無料枠が「20枚・5MBまで」と制限あり、Squooshは「1枚ずつ・設定が玄人向け」という不満が紹介記事で繰り返し指摘されている（cocotiie / on-ze等）。**この2大ツールの不満の隙間（一括×簡単×WebP既定）が狙い目**。
- **キャリアとの結びつき**: 黒田さんのMEMORYに「読者はpngは知っているがwebpは知らない層が多い」とある=まさにこのツールの教育的価値が刺さる読者層。ポートフォリオの「実用ツールを公開できる人」という証明にもなる。
- **技術土俵**: **静的（ブラウザ内完結）**。Canvas / `createImageBitmap` + WebP/AVIFエンコード（`@jsquash/webp` 等のWASM）。サーバー不要=コスト0・プライバシー訴求（画像を送信しない）が可能。kurodafolioの静的基盤と完全整合。
- **工数感**: **M（1〜2週）**。WASMエンコーダ組み込み・一括処理UI・ZIP化が山場。
- **競合との差別化**: (1)複数枚一括（TinyPNGの無料制限を超える）(2)WebP/AVIFを既定にした初心者向けUI（Squooshより簡単）(3)「画像を送信しない」プライバシー明記 (4)圧縮率の数値を大きく見せる"気持ちよさ"設計。
- **ブログ連動**: **HTML・CSS** または **SEO**（Core Web Vitals / 画像最適化）。「なぜWebPなのか」「ブラウザ内圧縮の仕組み（WASM）」で AI×Web制作 や JavaScript にも展開可能。

## A-2. OGPプレビューくん（仮）— OGP/Twitterカード入力→実寸プレビュー＆メタタグ生成

- **何をするツールか**: タイトル・説明・画像URL・サイト名を入力すると、X / Facebook / LINE / Slack でのカード表示を**実寸シミュレーション**し、コピペ用の `og:` / `twitter:` メタタグを生成。URLを入れたら既存OGPを読み取って検証する逆引きモードも。
- **想定ユーザー**: ①フロント・WordPress制作者（案件でOGP設定する全員）＋②ブロガー（SNSシェア時の見え方を事前確認したい）。
- **需要根拠**: 「OGP ジェネレーター」「OGP 画像 確認」は競合多数（web-toolbox / negi-lab等）＝需要証明済み。さらに紹介記事に「記事ごとに最適化したOG画像でシェア率向上・PV3倍」という訴求事例があり、ブロガーの関心が高い領域。
- **キャリアとの結びつき**: WordPress / SEO カテゴリの黒田さんの専門と直結。「OGP設定の解説記事→このツールで確認」という記事⇄ツール回遊が自然に組める。
- **技術土俵**: メタタグ生成・プレビューは**静的で可能**。ただし「URLを入れて既存OGPを読み取る逆引きモード」はCORSの壁があり、軽量プロキシ（**Cloudflare Workers**）が要る。**生成のみ=静的S / 逆引きまで=API要M**。段階リリース推奨。
- **工数感**: **S〜M**（生成プレビューのみS、逆引き込みM）。
- **競合との差別化**: (1)1画面で「生成」と「既存URL検証」両対応 (2)各SNSの最新カード仕様に実寸追従 (3)OGP画像のサイズ・比率も同時チェック。多くの競合は生成 or 確認の片方のみ。
- **ブログ連動**: **WordPress** / **SEO**。「XWriteのOGP設定」など既存運用ナレッジ（MEMORY: OGタイトル=タイトルタグ統一）とも噛み合う。

## A-3. スクロール演出ジェネレーター（仮）— CSSスクロール連動アニメをコード吐き出し

- **何をするツールか**: フェードイン・パララックス・進捗バー等のスクロール連動アニメをGUIで選び、パラメータ調整→**素のCSS（scroll-driven animations / `animation-timeline`）またはIntersectionObserver+CSSのフォールバック**をコピペ出力。
- **想定ユーザー**: ①フロント学習者・同業（LP制作で「ふわっと出る」演出を実装したい層は非常に厚い）。②は薄め。
- **需要根拠**: 「スクロールアニメーション CSS」は学習者の鉄板テーマ。native scroll-driven animations は2025〜2026で標準化が進む新領域で、Fixie Tools / FWD Tools 等の生成ツールが登場し始めた**ホットな端境期**。日本語の生成ツールはまだ手薄。
- **キャリアとの結びつき**: **黒田さんの既存資産と最強の相乗効果**。frontend-note には scroll-animation シリーズ（snippet004 parallax / 005 count-up 等）が既にあり、記事⇄ツールの回遊導線が即組める。LP制作という本業の腕の証明にも直結。
- **技術土俵**: **静的（ブラウザ内完結）**。プレビューとコード生成のみ。kurodafolio基盤と完全整合。
- **工数感**: **M**。プリセット設計とライブプレビューが山場。プリセットを絞ればSも可。
- **競合との差別化**: (1)**日本語**で native scroll-driven animations を扱う先行者になれる (2)フォールバック込みで「実務でそのまま使える」コードを出す（海外ツールは最新CSS前提で実務に落としにくい）(3)既存シリーズ記事という解説資産がある。
- **ブログ連動**: **HTML・CSS** / **JavaScript**。既存 scroll-animation シリーズの「決定版ツール」として位置づけられる。

## A-4. 開発お助けボックス（仮）— よく使う変換系を1ページに束ねたマルチツール

- **何をするツールか**: JSON整形/圧縮・Base64・URLエンコード・正規表現テスト・文字コード変換・diff など、開発でちょい使いする変換を**1ページにタブで集約**。全てブラウザ内処理。
- **想定ユーザー**: ①フロント・バック問わず開発者全般。②は薄い。
- **需要根拠**: JSONフォーマッタ・正規表現テスター・Base64等は個別では超レッドオーシャン（rakko / web-toolbox / oh-benri 等が網羅的に提供）＝**需要は確実だが単機能では埋もれる**。勝ち筋は「束ね方とUXの良さ」。
- **キャリアとの結びつき**: 開発者の常用ツールになれば被リンク・ブックマーク資産になる。ただし「黒田さんらしさ（フロント/LP）」は出にくく、実力証明としては弱め。
- **技術土俵**: **静的（ブラウザ内完結）**。
- **工数感**: **M〜L**（機能数で変動。1機能ずつ追加できるので最小構成Sスタート可）。
- **競合との差別化**: 正直**差別化は難しい領域**。勝てるとすれば(1)動作の速さ・広告の少なさ・UIの洗練 (2)「全部1ページ・タブ切り替え不要級の快適さ」。コモディティ前提で、第1弾より「常用資産の積み増し」枠。
- **ブログ連動**: **JavaScript** / **AI×Web制作**（「マルチツールをどう設計したか」の裏側記事）。★要確認: 差別化が弱いため第1弾優先度は低めと判断。

## A-5. カラーパレット＆コントラストくん（仮）— 配色生成＋アクセシビリティ判定

- **何をするツールか**: ベースカラーから配色パレットを自動生成し、各組み合わせの**WCAGコントラスト比（AA/AAA合否）**を同時表示。CSS変数・Tailwind config でコピー出力。
- **想定ユーザー**: ①フロント（黒田さんの強み=アクセシビリティ重視と直結）＋②デザイナー・ノンデザイナー（配色に自信がない制作関係者）。両層に効く。
- **需要根拠**: CSSジェネレータ系のまとめ記事（sdesignlabo 17選 / cohamu 12選等）で配色・コントラストは定番枠＝需要証明済み。コントラスト判定は近年アクセシビリティ義務化文脈（改正障害者差別解消法）で関心上昇。
- **キャリアとの結びつき**: **「保守性・アクセシビリティ重視のクリーンなコーディング」という黒田さんの会社強みを体現するツール**。ポートフォリオでの実力証明とブランド整合が高い。
- **技術土俵**: **静的（ブラウザ内完結）**。色計算のみ。
- **工数感**: **S〜M**。
- **競合との差別化**: (1)生成（配色）と検証（コントラスト）を**1画面で往復**できる（多くは別ツール）(2)CSS変数/Tailwind 即出力で実装まで地続き (3)アクセシビリティ文脈の解説を黒田さんの強みとして添える。
- **ブログ連動**: **HTML・CSS** / **SEO**（アクセシビリティ）。黒田さんの強みカテゴリで深い記事が書ける。

## A-6. コピペ用 余白＆タイポ整えくん（仮）— 8pxグリッド/タイポスケール計算

- **何をするツールか**: ベースフォントサイズと比率を入れると、見出し〜本文のタイポスケール（modular scale）と8pxベースの余白セットを生成し、CSS変数 / clamp() レスポンシブ指定でコピー出力。
- **想定ユーザー**: ①フロント学習者（余白・サイズ設計に悩む駆け出し）＋②ノンデザイナー制作者。
- **需要根拠**: 「タイポスケール」「8pxグリッド」「clamp 計算」は学習者の頻出悩み。海外に Type Scale 等の先行ツールがあり需要は実在。日本語＆clamp対応は手薄。
- **キャリアとの結びつき**: 設計の地に足ついた感=黒田さんの「設計重視」ブランドと整合。ただしニッチで話題性は中程度。
- **技術土俵**: **静的（ブラウザ内完結）**。
- **工数感**: **S**。
- **競合との差別化**: (1)余白とタイポを**同時に1セット**で出す (2)`clamp()` レスポンシブ前提（海外ツールは固定値が多い）(3)日本語フォント前提の行間も配慮。
- **ブログ連動**: **HTML・CSS**（レスポンシブ・clamp 解説）。

## A-7. なんでも文字カウント＆原稿チェッカー（仮）— 文字数＋SEO向け補助

- **何をするツールか**: リアルタイム文字数/行数/原稿用紙換算に加え、**SEO/SNS向け補助**（タイトル◯字超過警告・meta description長チェック・X140字判定・全角半角別カウント）を1画面で。
- **想定ユーザー**: ②ブロガー・ライター・SNS運用者（コードを書かない層）。①にはやや薄い。
- **需要根拠**: 「文字数カウント 無料」はラッコ等が定番化＝需要は超太い。一方で大半が「ただ数えるだけ」。SEO/SNSの文字数制約チェックを束ねた版は差別化余地あり。
- **キャリアとの結びつき**: SEO・WordPressカテゴリの黒田さんと接点あり。非エンジニア層への入口として広いが、実力証明性は低い。
- **技術土俵**: **静的（ブラウザ内完結 / 入力をサーバー送信しない安心訴求）**。
- **工数感**: **S**。
- **競合との差別化**: (1)カウントに留めず**SEOタイトル/meta/X**の文字数ルールを同居 (2)「サーバーに送らない」明記。コモディティ寄りなので単独より「ブロガー導線の入口」枠。
- **ブログ連動**: **SEO** / **学習・仕事術**。

# カテゴリB: キャリア周辺ツール（差別化・第2弾以降）

> 黒田さんの専門性が滲むが、より広い層にも届くもの。

## B-1. LP診断くん（仮）— URLを入れるとLPの実装チェックを自動診断

- **何をするツールか**: URLを入力すると、画像のWebP化率・alt欠落・見出し階層・OGP有無・表示速度の簡易指標・レスポンシブ崩れ兆候などを**チェックリスト形式でスコア表示**し、改善提案を出す。
- **想定ユーザー**: ②個人事業主・ブロガー（自分のサイトの問題を知りたいが専門知識がない層）＋①駆け出し（自作LPのセルフレビュー）。
- **需要根拠**: PageSpeed Insights等の「URL入れて診断」型は強い需要があるが、初心者には数値が難解。**「フロントエンジニア視点の噛み砕いた指摘」という切り口は競合が薄い**。
- **キャリアとの結びつき**: **案件獲得に最も直結する候補**。「あなたのサイト、ここが惜しい」→「直しましょうか?」という自然な営業導線。ポートフォリオの実力証明としても最強格。
- **技術土俵**: **API要（Cloudflare Workers でクロール＆解析）**。CORS・速度計測のためバックエンド必須。kurodafolioのAPI許容方針が活きる代表例。
- **工数感**: **L**。クロール・解析ロジック・誤検知対策が重い。
- **競合との差別化**: (1)数値羅列でなく**初心者向けの平易な改善提案** (2)フロント実装者ならではのチェック項目（alt/見出し/WebP）(3)診断→相談への導線。
- **ブログ連動**: **SEO** / **AI×Web制作**（診断ロジックの裏側）。★要確認: 工数Lかつ誤診のブランドリスクがあるため、第1弾ではなく実績を積んでから。

## B-2. メタタグ一括生成くん（仮）— SEO/OGP/構造化データをまとめて出力

- **何をするツールか**: サイト情報を入力すると、title / description / OGP / Twitterカード / 基本的なJSON-LD構造化データ（Article / BreadcrumbList / FAQPage）を**まとめてコピペ出力**。
- **想定ユーザー**: ①WordPress/フロント制作者＋②自分で多少いじれるブロガー。
- **需要根拠**: 構造化データは黒田さんが既に記事化済み（MEMORY: JSON-LD記事 / FAQPage・BreadcrumbList実装ハンズオン）＝**記事資産があり需要も把握済み**。手書きは面倒で需要が太い。
- **キャリアとの結びつき**: SEO/WordPressの専門記事と強連動。既存記事の「実践ツール」化。
- **技術土俵**: **静的（ブラウザ内完結）**。
- **工数感**: **M**。スキーマ種別ごとの入力フォーム設計が山場。
- **競合との差別化**: (1)OGPとJSON-LDを**1ツールで**（多くは分離）(2)黒田さんの構造化データ解説記事という権威付け (3)コピペで即WordPressに貼れる粒度。
- **ブログ連動**: **SEO** / **WordPress**。A-2(OGP)とも統合発展しうる。

## B-3. クライアント向け かんたん見積りシミュレーター（仮）

- **何をするツールか**: 「LP / コーポレート / WordPress」「ページ数」「アニメーション有無」等を選ぶと、**目安の制作費レンジと納期**を提示。問い合わせフォームに接続。
- **想定ユーザー**: ②発注を検討する個人事業主・中小（黒田さんの見込み客そのもの）。
- **需要根拠**: 「ホームページ制作 費用 相場」は検索需要が非常に太い定番テーマ。見積りシミュレータを置く制作会社・フリーは多く、リード獲得手段として実績ある型。
- **キャリアとの結びつき**: **直接の集客装置**。ポートフォリオ層と最も相性が良い（kurodafolioの作品紹介→見積り→問い合わせ）。
- **技術土俵**: **静的（計算ロジックのみ）＋ 送信に SSGForm 等**（kurodafolioで既に実績のあるフォーム送信基盤を流用可）。
- **工数感**: **S〜M**。価格ロジック（黒田さんの実単価）の設計が要相談。★要確認: 価格を公開する是非・レンジ設定は黒田さんの経営判断。
- **競合との差別化**: (1)誠実トーン（盛らない見積り＝ブランド「静かな持続可能性」と整合）(2)実装者本人が答える安心感。
- **ブログ連動**: **学習・仕事術**（フリーの価格設定論）。やや内向き。

## B-4. コードハイライト画像メーカー（仮）— コードを綺麗な画像にしてSNSシェア

- **何をするツールか**: コードを貼ると、シンタックスハイライト済みの**おしゃれな画像（OG比率/正方形）**を生成。X/Qiita/note のシェア用。テーマ・背景・言語選択可。
- **想定ユーザー**: ①エンジニア全般（技術発信する人）＋②デザイン意識の高いブロガー。
- **需要根拠**: 海外の Carbon / ray.so が大ヒットしている領域＝**需要は世界的に証明済み**。日本語UI＆日本語フォント綺麗対応はまだ手薄。

# 第1弾として推す3案（優先順位付き）

> 選定軸: **実用性・話題性 × 現実的工数 × キャリア寄与**のバランス。
> 実力証明・SEOは第1動機にしない。「使われる・シェアされる・被リンクされる」を最優先しつつ、工数が現実的で副次的にキャリアにも効くものを選定。

### 第1位: A-1 ぱっと画像最適化くん（Web用画像 一括圧縮＆WebP変換）

- **理由**: 需要が最も太く実証済み（TinyPNG/Squoosh級のレッドオーシャン＝裏返せば最大需要）。かつ2大競合の**明確な不満（一括不可 / UI玄人向け）に刺さる差別化が立つ**。①②両ターゲットに同時に効く唯一の案。静的・WASMで完結しコスト0、工数M。MEMORYの「読者はWebPを知らない」層への教育的価値も乗る。話題性・被リンク適性・実用性が三拍子そろう本命。

### 第2位: A-3 スクロール演出ジェネレーター（CSSスクロール連動アニメ生成）

- **理由**: native scroll-driven animations が標準化する**端境期で、日本語の生成ツールが手薄＝先行者利益が狙える**。何より frontend-note の既存 scroll-animation シリーズという**記事資産との回遊が即組める**のが決定的。同業・学習者に強く刺さりSNS拡散適性も高い。静的完結・工数M。本業LP制作の腕の証明も兼ねる。

### 第3位: A-5 カラーパレット＆コントラストくん（配色生成＋アクセシビリティ判定）

- **理由**: 工数がS〜Mと最も軽く**早期に出せる**。需要は定番枠で実証済み、①フロント＋②ノンデザイナー両層に届く。黒田さんの会社強み「アクセシビリティ重視のクリーンな設計」を体現し、ブランド整合が3案中最高。アクセシビリティ義務化の追い風もあり、解説記事との連動も深い。

**補足**: B-1(LP診断くん)とB-3(見積りシミュレーター)は**キャリア・案件獲得への寄与が最大級**だが、前者は工数L＋誤診リスク、後者は価格公開の経営判断が絡むため第1弾から外した。第1弾で実用ツール公開の実績とトラフィックを作った後、第2弾で投入するのが堅実。

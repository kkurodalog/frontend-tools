# 02. アーキテクチャ決定支援 — 共通部品「共有」vs「独立（コピー）」（保守性分析）

担当: Cody（制作部 / コーディング・技術レビュー・品質点検）
作成日: 2026-06-17
対象: 黒田さん相談「kuroda-tools の CSS・JS を本体 kurodafolio.com と共有すべきか、今のまま別サイトのように独立構築すべきか。保守性優先で検討」
位置づけ: 決定支援ドキュメント（正本候補）。実装変更は本ドキュメントでは行わない（黒田さんの判断待ち）。
関連: `00_tech-verification.md` §③（partial 同期＝コピー運用推奨）／`PLAN.md` §2-5（独立 Vite・本体触らない原則）

---

## 0. 結論（先出し）

**現状の「独立構築（コピー運用）」を継続するのが保守性の観点で明確に有利。**「共有」は今のリポ構成では保守を楽にするどころか、結合コスト・機密リスク・ビルク複雑化を新たに生む。

ただし**1点だけ将来の折衷余地**として「**デザイントークンの値だけを単一ソース化（軽量パッケージ or JSON 共有）する**」案は検討に値する。とはいえ現時点では不要——後述のとおり、トークン改定の頻度が「独立コピーの手追従コスト」を上回るまで温めておけばよい。

`00_tech-verification.md` §③（partial＝コピー運用推奨）の既存結論は **支持・拡張する**（partial だけでなく CSS・JS・トークンを含めた全レイヤーで独立が妥当）。

---

## 1. 「共通部品を共有する」とは技術的に何を意味するか（選択肢の分解）

「共有」は曖昧語なので、実現手段ごとに分解して実現性を評価する。

| # | 方式 | 具体 | 実現性 | 評価 |
|---|---|---|---|---|
| ① | **npm パッケージ化** | 共通 UI（header/footer/トグル）と tokens を `@kuroda/ui` 等の npm パッケージにし、本体と kuroda-tools の両リポが依存 | 技術的に可能だが、本体は SCSS・tools は素 CSS でビルド系が違う＝パッケージ側で両対応の配布形態（コンパイル済み CSS + 生 SCSS + CSS 変数）を整える重い作業が要る。publish/version 運用も発生 | △ オーバーエンジニアリング |
| ② | **git submodule** | 共通部品ディレクトリを別リポにし、両プロジェクトが submodule で取り込む | 可能。ただし submodule は更新・clone・CI で事故りやすく、公開リポ（tools）が非公開の本体由来 submodule を参照すると公開境界が壊れる | ✕ 公開×非公開で破綻 |
| ③ | **monorepo 化** | 本体と tools を 1 リポに統合し packages を共有 | **PLAN §2-5「別リポ・別ビルド・別デプロイ」と、本体＝非公開／tools＝公開の境界に正面衝突**。統合は機密混入リスクが最大 | ✕ 却下 |
| ④ | **デザイントークンだけ共有** | 色・余白・タイポの CSS 変数（値）だけを単一ソース化（JSON or 小さな CSS ファイルを npm/コピー配布） | 部品（HTML/JS 構造）は共有せず、値だけを同期。最も結合が薄い | ○ 唯一の将来折衷候補 |
| ⑤ | **ビルド時に本体ソースを参照** | tools のビルドが本体リポの SCSS/partial を直接 import | 別リポ・別ビルドのため相対参照が成立しない（パス・依存・SCSS↔素CSS 差）。本体を clone しないと tools がビルドできなくなる＝**独立性の完全喪失** | ✕ 却下 |
| ⑥ | **現状のコピー運用（独立所有）** | tokens は本体値を 1:1 コピー。partial・トグルは本体実装を移植してツール側で所有 | 既に稼働中。追加コストゼロ | ◎ 現状維持 |

**結論**: 現実的に検討に値するのは ④（トークンのみ）と ⑥（現状）の二択。①②③⑤ はいずれも公開×非公開リポ境界・SCSS↔素CSS差・別ビルド/別デプロイという既存前提と衝突し、保守性をむしろ悪化させる。

## 2. 公開リポ（パターンA）× 非公開リポ（パターンB）を跨ぐ共有の問題

ここが本件の核心。kuroda-tools と本体は単に「別プロジェクト」なだけでなく、**公開境界が逆**である。

| 軸 | kuroda-tools（パターンA） | kurodafolio 本体（パターンB） |
|---|---|---|
| リポ公開性 | **GitHub Public**（全ファイル公開前提） | **非公開**（クライアント案件構造・機密多数） |
| ビルド | 独立 Vite（`base:"/tools/"`）・**素 CSS** | 別ビルド・**SCSS** |
| デプロイ単位 | `public_html/tools/` 配下 | 静的サイト・`public_html/` 直下（XServer） |
| my-virtual-team | 除外（`projects/_self/.gitignore` の `*`） | 履歴管理（例外規定） |

### 2-1. 機密混入リスク（最重大）
本体は「クライアント案件構造」を持つ非公開リポ。そのソース（SCSS／partial／設定）を**公開リポの tools から submodule/参照/同梱すると、公開リポに本体ソースが流れ込む経路ができる**。共有 partial に将来クライアント名・社内 URL・未公開情報が紛れれば、それが GitHub Public に露出する。これは保守性以前の**セキュリティ事故リスク**であり、共有方式（②③⑤）の決定的な欠点。④（トークン値のみ）なら漏れるのは色・余白の数値だけなので、この観点では唯一許容できる。

### 2-2. SCSS vs 素 CSS のビルド差
本体は SCSS（変数・mixin・ネスト）。tools は素 CSS（CSS カスタムプロパティ）。SCSS の partial をそのまま tools で使うには tools 側に SCSS ツールチェーンを足す必要があり、「素 CSS で軽量」という tools の設計を崩す。逆に CSS 変数（`--color-*` 等）は**両者で共通言語**なので、共有するなら「コンパイル済みの値（CSS 変数）」レイヤーが唯一の接点になる＝④に帰着する。

### 2-3. 別デプロイ
本体は XServer 直下の静的サイト、tools は `/tools/` 配下の別 dist。デプロイ単位が分かれているため、「共有部品を直したら両方が同時に更新される」自動追従は**そもそもデプロイ段階で分断**される（片方だけ再デプロイすればバージョン非同期が起きる）。共有の最大メリット（自動追従）が、別デプロイ前提では実は効きにくい。

## 3. 保守性の実態比較（忖度なし）

「共有＝保守が楽」は直感的だが、本件の前提では成り立たない。

| 観点 | 共有（②③⑤）が生むコスト | 独立（⑥）の実コスト |
|---|---|---|
| 本体変更の追従 | 自動追従するが、**追従してほしくない変更まで巻き込む**（本体の内部ナビ刷新が tools の逆導線を壊す等。そもそも役割が逆＝§3-1） | 本体のロゴ/ブランド変更時のみ手で追従。**頻度は年に数回レベル**・対象は `vite.config.js` の `context.site`（本体URL/ブランド名）1 箇所に集約済み |
| 破損リスク | 共有部品の破壊的変更が**両サイト同時に巻き込む**。tools のリリースが本体改修に人質に取られる | 本体側の変更が tools ビルドを壊さない（完全分離） |
| 機密 | 公開リポに本体機密が混入する経路ができる（§2-1） | 経路が存在しない |
| ビルク複雑化 | SCSS↔素CSS のブリッジ・パッケージ publish・submodule 運用が増える | 追加ツールチェーンなし（素 CSS のまま） |
| バージョン同期 | 別デプロイのため結局手で揃える必要（§2-3）＝自動追従の旨味が薄い | 各々独立リリースで非同期が問題にならない |

### 3-1. 「役割が逆」という設計事実（00_tech-verification §③ の再確認）
本体の header/footer は**内部ナビ**（About/Works へ回遊させる）。tools の header/footer は**本体への逆導線＋パンくず**（Home > Tools > {ツール名}）＝営業ファネルの向きが逆。**そもそも同一であるべきでない部品**を共有すると、差を吸収する条件分岐が partial に増えて逆に複雑化する。これは「重複」ではなく「役割の違う別物」であり、コピー＝独立所有が情報設計上も正しい。

### 3-2. 独立（コピー）の重複コストの実規模
唯一の弱点は「本体 UI 変更時に手で追従」。だが実際の追従対象は:
- **トークン値**（色・余白・タイポ）＝本体のリブランドが起きたときだけ。頻度は低い。
- **本体への絶対URL・ブランド名**＝既に `context.site` 1 箇所に集約済み（00_tech-verification §③ 緩和策）。

つまり重複コストは「年数回・1〜数箇所の手編集」程度で、共有が生む結合コスト（機密・ビルド・破損・同期）より**桁違いに小さい**。

## 4. 本体「触らない原則」（PLAN §2-5）との整合

PLAN §2-5 は「kurodafolio 本体は触らない（導線追加という最小変更を除く）」を原則化している。

- **共有方式（②③⑤）**: 共有部品の置き場所・所有者を本体側に持つ／本体ビルドへ依存する形になりやすく、**本体への変更・本体ビルドへの依存を新たに生む**＝触らない原則に反する。
- **独立（⑥）**: 本体に一切触れず完結。原則と完全整合。
- **トークンのみ共有（④）**: 配布形態次第。本体側を「ソース」にすると原則に触れるが、**第三のニュートラルな小リポ／パッケージ**に値だけ切り出せば本体非依存を保てる。ただしその小リポの新設・運用が新コスト。

## 5. 推奨と段階論

### 5-1. 推奨（現時点）
**⑥ 独立構築（コピー運用）を継続。** 理由を保守性順で:
1. **結合コスト ≫ 重複コスト**: 共有が生む機密混入・ビルド複雑化・破損連鎖・バージョン非同期は、独立の手追従コスト（年数回・1箇所集約済み）を大きく上回る。
2. **公開×非公開の境界を侵さない**: 公開リポに本体機密が流れる経路をそもそも作らない。
3. **役割が逆の部品**: header/footer は本体（内部ナビ）と tools（逆導線＋パンくず）で役割が異なる別物。共有は条件分岐を増やすだけ。

### 5-2. 段階論（将来トリガー）
今は不要だが、以下が起きたら **④ トークンのみ単一ソース化** を再検討する:
- 本体のリブランド（色・タイポ改定）が**年複数回**の頻度になり、tools への手追従が負担化したとき。
- その場合も共有するのは**CSS 変数の値だけ**（JSON or 小 CSS をニュートラルな配布元から両者にコピー/取得）。**HTML 構造・JS・partial は共有しない**（役割差・機密境界のため）。

### 5-3. 折衷の率直な評価
「トークンだけ将来パッケージ化、部品はコピー継続」は理にかなった折衷だが、**現状のトークン改定頻度では割に合わない**（パッケージ運用の固定コスト > 手追従の変動コスト）。よって今は ⑥ のまま、④ はトリガー発火時の選択肢として温存が最適。

## 6. U-02 ③（既存結論）との関係

`00_tech-verification.md` §③ は「partial 同期＝(A) コピー運用（独立所有）推奨」を結論済み。本分析は**これを覆さず、むしろ CSS・JS・デザイントークンを含む全レイヤーへ拡張して支持**する。§③ が partial 単体の結論だったのに対し、本ドキュメントは「kuroda-tools 全体のスタイル/スクリプト基盤を本体と共有すべきか」という上位問いに対して同じ結論（独立）を、公開×非公開リポ境界・SCSS↔素CSS差・別デプロイという追加観点で補強した。

## 検証済み
- **検証済み: 事実ベース** — 公開（パターンA）／非公開（パターンB）リポ区分・別 Vite ビルド・別デプロイ（XServer `/tools/` vs 直下）・本体 SCSS／tools 素 CSS・tokens は本体値の 1:1 コピー（`foundation/_tokens.css` 冒頭「実値で継承」明記）を反映。PLAN §2-5「触らない原則」・00_tech-verification §③ と整合。
- **検証済み: 推測と事実の区別** — 「トークン改定頻度が低い」「リブランドは年数回」は**推定**（実績データなし）として明記。リポ構成・ビルド差・デプロイ分離は**事実**。共有方式の機密混入リスクは公開×非公開境界からの**論理的帰結**。
- **検証済み: 結論の明確性** — 保守性観点で「独立継続（⑥）」を推奨と明示。折衷（④）は将来トリガー付きで率直に位置づけ。U-02③ を覆さず支持・拡張。

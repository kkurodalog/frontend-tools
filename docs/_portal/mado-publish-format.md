## 0. 結論（先出し）

**推奨 = 案B（ツールポータル型）。ただし案Cの思想を「ツール単位ではなくポータル単位」で取り込む折衷。**

- **配置**: 単一サブドメイン `tools.kurodafolio.com` を「黒田さんのフロントツール集」ポータルにする。トップ=ツール一覧、各ツール=**サブディレクトリ** `tools.kurodafolio.com/image-compress/` 形式。
- **既存「1ツール=1サブドメイン」前提は破棄**する（ドメイン評価が分散し、個人事業では保守も重い。`mado-scope-decision.md §3-2` の結論＝「乱立せず1拠点に集約」と完全に整合）。
- **第1弾の出し方**: 画像仕分け圧縮くんを**ポータルの最初の1ツールとして**公開する（単独先行ではない）。トップのツール一覧に1枚だけカードが並ぶ状態でローンチし、以後clamp・配色・FLOCSSをカードとして足していく。
- **案C（機能を1アプリに束ねる）は不採用**。画像圧縮とclampは利用シーンが別で、束ねるとUIが過密になり各検索意図への最適化も薄まる。**「束ねる」のはアプリ内機能ではなく、ポータルというハコの中の独立ページ群**として実現する。

一言理由: 各ツールは別々の検索意図（「画像圧縮」「clamp 計算」「配色」）を持つので**1ツール1ページで個別最適**しつつ、それらを**1ドメインに集約**してドメイン評価・内部リンク・回遊を1拠点に貯める——これがmach-tools/ラッコが実際に取っている勝ち筋。

## 1. 実例調査（Web検索で裏取り）

### 1-1. mach-tools.net（黒田さんが見つけたサイト）
- **ポータル型・単一ドメイン・サブディレクトリ**構成を実機確認。トップは「公開中56ツール / 開発中26ツール」のツール一覧。
- カテゴリ6分類（仕事・事務 / Web・制作 / 教育・発想 / 地図・検索 / 健康・運転 / 生活・便利）。
- 各ツールのhref実例: `/password/`、`/qrcode/`、`/colorpalette/` ——**すべて同一ドメイン直下のサブディレクトリ**。サブドメインは使っていない。
- → 1ツール=1ページを多数並べ、1ドメインに束ねる典型。**今回の案Bそのもの**。

### 1-2. ラッコツールズ（rakko.tools / 国内最大級・月間150万PV実績）
- **サブディレクトリ形式**を実機確認。ツールURLは `rakko.tools/tools/91/`・`/tools/2/`・`/tools/4/` ——全ツールが**同一ドメイン配下**。サブドメインで各ツールを分けていない。
- 130種類以上のツールを、SEO/SSL/Text/Network/Design等**約20タグでカテゴリ分け**。トップは人気ツール5つを先頭に出すポータル型。
- 別途 `rakko.tools/workshop/`（ツール解説ブログ）を同一ドメイン配下に持ち、**ツール×記事の回遊**を1ドメイン内で完結させている。
- → 国内最成功のツール集が「サブドメイン乱立」ではなく「1ドメイン＋サブディレクトリ＋タグ分類＋解説記事の同居」で巨大流入を得ている実証。

### 1-3. ポータル型がSEOを伸ばす仕組み（裏取り）
- ポータルサイトのSEOは「**カテゴリ（一覧）ページに評価を集約する全体最適**」「正規化と**内部リンクの集中設計**」が成果の前提（lany / NY-marketing）。
- 多数ページが**内部リンクで相互接続**され回遊性が上がる→滞在時間延長→ドメイン評価が育つ（Hub Works / SEOホッパー）。
- → ツールポータルは「各ツールページが個別キーワードを取りに行く入口」かつ「1ドメインに被リンク・流入・回遊を集中させる器」として機能する。**ツールが増えるほど1ドメインの評価が複利で育つ**構造。

## 2. SEOの観点（最重要）

### 2-1. サブドメイン vs サブディレクトリ
- **Google公式見解**: サブドメインとサブディレクトリに直接の優劣はない（順位アルゴリズム上はどちらでも可）。
- **ただし実務上の差は明確**: サブドメインは**別サイト扱いされやすく評価がそのサブドメインに留まる**。被リンクを得てもメインドメインのパワー強化に繋がりにくく、**新サブドメインはゼロから評価を積み直す**。サブディレクトリは**メインドメインのパワーを引き継ぎ、評価を1ドメインに集約**できる（stock-sun / wacul / digitalift 各社一致）。
- **本件への含意**: ツールが増える前提だと、`{tool}.kurodafolio.com` で1つずつサブドメインを立てる＝**ツールごとに評価ゼロスタートが何個も発生**し、ドメインパワーが分散する。逆にサブディレクトリ集約なら、**画像圧縮ツールが得た被リンクがclampツールにも効く**＝ドメイン全体が複利で育つ。**ツールが多いほどサブディレクトリ集約が有利**。

★補足: 本体 `kurodafolio.com` はポートフォリオ（営業の顔）。そこにツールのサブディレクトリを同居させると本体の性格が「ツール集」に寄り、ポートフォリオの訴求が薄まる懸念がある。そこで**ツール群専用に1サブドメイン `tools.kurodafolio.com` を立て、その配下をサブディレクトリで束ねる**のが最適解。本体ポートフォリオとツール集を性格分離しつつ、ツール群内部では評価を集約できる（サブドメイン1個分の「ゼロスタート」は許容し、それ以降は配下に集約する設計）。

### 2-2. 「ポータル多数ページ」vs「単一ページ」vs「束ねた1アプリ」の検索カバレッジ比較

| 形式 | 検索カバレッジ | ドメイン評価蓄積 | 判定 |
|---|---|---|---|
| **案A 単一ページ（1ツール=1サブドメイン）** | 各ツールが個別キーワードで最適化できる（◎）が、サブドメインごとに評価分散（×）。ツールが増えるほど分散が累積 | 分散。1つ伸びても他に波及しない | △ 個別最適は良いが評価が育たない |
| **案B ポータル（1ツール1ページ×多数を1ドメイン集約）** | 各ツールページが個別検索意図を**個別最適**できる（◎）。かつ被リンク・回遊が1ドメインに集約（◎） | **複利で蓄積**。ツールが増えるほど強くなる | **◎ 最良。個別最適と評価集約を両立** |
| **案C 機能を1アプリに束ねる（A-4型）** | 1ページに画像圧縮もclampも配色も同居→**どの検索意図にも中途半端**。「画像圧縮」で来た人向けの専用ページが存在しない（×） | 1ページに集中はするが、各キーワードで上位を取れない | × カバレッジ最悪 |

- **核心**: 各ツールは**別々の検索意図**を持つ（「画像圧縮 ツール」「clamp 計算」「配色 コントラスト チェック」）。案Cで束ねると、1つのページタイトル・1つのmeta・1つのh1に複数意図を詰め込むことになり、**どの意図にも最適化が刺さらない**。案Bなら `/image-compress/` は画像圧縮意図に、`/clamp/` はclamp意図に、それぞれ専用ページとして個別最適でき、**検索カバレッジが意図の数だけ広がる**。
- → **流入最大化の観点では案B一択**。案Cは「分けるべき検索意図を統合してカバレッジを自ら潰す」最悪手。

## 3. 利便性・回遊・ブランドの観点

- **ポータルの回遊効果**: ラッコの実例どおり、「画像圧縮を使いに来た人が、トップやサイドのツール一覧で clamp/配色 にも気づく」相互送客が働く。単一ページ（案A）はこの回遊がゼロ＝来訪者が1ツールで離脱。**ツール集として育てるなら回遊資産は大きい**。
- **単一ページの「分かりやすさ」効果**: 案Aの利点（1用途に集中して迷わない）は、**案Bでも各ツールページ単位で完全に確保される**。ポータルでも個別ツールページは「画像圧縮だけやる潔いページ」のままだから、案Aの長所は案Bに内包される。失うものがない。
- **destinationブランドの価値**: 「黒田さんのフロントツール集＝ここに来ればフロント制作の道具が揃う」という拠点ブランドは、`mado-scope-decision.md §3-2` で既に「動きを作るならここ、の1拠点に集約」と評価した思想の延長。ツールが散らばる案Aより、**1拠点に集約する案Bの方がブランド資産・記事連動・被リンク先として強い**。黒田さんのブログ（FLOCSS/clamp/画像最適化の記事資産）からツール集へ送客し、ツール集からブログへ送り返す回遊も1拠点だから組める。
- **案C（束ねる1アプリ）の使い勝手**: 画像圧縮（画像をD&Dして書き出す作業）とclamp（数値を入れて値を作る作業）は**利用シーン・利用タイミングが完全に別**。束ねる必然性がない。1アプリにタブで同居させると、UIが過密になり、初学者が「自分に関係ないタブの森」で迷う（`mado-scope-decision.md §2` のプログレッシブ・ディスクロージャー論と同じ弊害）。**束ねる便益＜過密のコスト**。

## 4. 運用・拡張・既存構造との整合

### 4-1. 運用負荷
- **案A（サブドメイン乱立）**: ツール数だけ別サブドメイン＝ビルド・デプロイ・SSL・保守・更新が増殖。個人事業では重い（`mado-scope-decision.md §3-2` の結論と同じ）。
- **案B（1コードベース集約）**: 1リポジトリ・1ビルド・1デプロイで全ツールを管理。ツール追加=配下にページを1つ足すだけ。**保守が集中して軽い**。
- → 運用面でも案Bが明確に有利。

### 4-2. 既存「1ツール=1サブドメイン」前提の見直し → **見直す（破棄する）**

**明言: 当初の「層2 = `{tool}.kurodafolio.com` / 1ツール1サブドメイン」前提は破棄し、「層2 = `tools.kurodafolio.com` ポータル + 配下サブディレクトリ」に改める。**

理由:
1. SEO（§2-1）: サブドメイン乱立はツールごとに評価ゼロスタートで分散。サブディレクトリ集約なら複利で育つ。ツールが増える前提ほど差が開く。
2. 運用（§4-1）: 乱立は保守増殖、集約は保守集中。
3. ブランド/回遊（§3）: 集約の方がdestinationブランドと相互送客を作れる。
4. **`mado-scope-decision.md §3-2` で既に「別サブドメイン乱立 vs 1ツール統合」を比較し『1拠点集約が運用/SEO/ブランドすべてで有利』と結論済み**。今回の検討はそれをツール群全体に拡張して同じ結論に到達＝判断が一貫している。

### 4-3. `motion.kurodafolio.com`（スクロールツール用）の扱い

`mado-scope-decision.md` では、スクロールツールに中立語サブドメイン `motion.kurodafolio.com` を割り当て、「将来アニメ全般化してもURL据え置きで資産継承」とした。今回のポータル化で、これを以下のどちらに寄せるか整理する。

- **方針（推奨）**: スクロールツールも**ポータル配下に寄せる** → `tools.kurodafolio.com/motion/`（または `/scroll/`）。理由: §2-1の集約メリットを取り、ツール群を1拠点に統一できる。`motion.kurodafolio.com` 単独サブドメインだと、また1個ゼロスタートのサブドメインが増えてしまう。
- **★要確認**: ただしスクロールツールは「内部にカテゴリを足して育てる1アプリ（motion=動き全般の拠点）」という固有のブランド構想を持つ（`mado-scope-decision.md §5-3`）。これは**ポータル内の1ツールページとしても成立する**（`/motion/` ページ自体がカテゴリ切替UIを持てばよい）ので、**サブディレクトリ化で支障はない**。`motion.kurodafolio.com` という独立サブドメインに固執する必要はなくなった、というのが今回の更新。スクロールツールがまだ未着手・温存中の今なら、URL確定前なので**ポータル前提に寄せるのが最も損が少ない**。

### 4-4. static-template（Vite + Handlebars）でのポータル実装イメージ
- **トップ（`/`）= ツール一覧ページ**: ツールをカード（サムネ＋名前＋一言説明＋カテゴリタグ）で並べる。カードのリンク先が各ツールページ。mach-tools/ラッコと同じ。
- **各ツール（`/image-compress/`, `/clamp/`, `/color/` …）= 配下ページ**: Handlebars の partial で「共通ヘッダー（ポータルへ戻る／他ツールへの導線）＋ツール本体」を組む。共通レイアウトを partial 化すれば、新ツール追加は「ページHTML1枚＋ツール固有JS」を足すだけ。
- FLOCSS: ツールカード= `.c-tool-card`、一覧= `.p-tool-index`、各ツール本体= `.p-tool-{name}` のように層分離。
- Vite のマルチページ構成（rollup input に各ツールHTMLを列挙）で、1ビルドで全ツールを吐ける。**1コードベース＝1ビルドで運用集約**（§4-1）が技術的に自然に実現できる。
- 各ツールページに「他のツール」内部リンクブロックを置けば、§1-3の内部リンク集中・回遊が設計で担保できる。

## 5. 結論：推奨フォーマットと配置方針

### 5-1. 推奨 = **案B（ツールポータル型）／折衷で案Cの思想は「ハコ単位」で吸収**

- 案A（単一/サブドメイン乱立）: 評価分散・運用増殖で不採用。
- **案B（ポータル・サブディレクトリ集約）: 採用**。個別最適（検索カバレッジ）と評価集約（ドメインパワー複利）を両立する唯一の案。
- 案C（機能を1アプリに束ねる）: 検索カバレッジを自ら潰し、UIも過密で不採用。ただし「ツールを1箇所に束ねる」という思想自体は正しく、それは**アプリ内機能の統合ではなく、ポータルという器への集約**で実現する＝これが折衷。

### 5-2. URL設計（具体形）

```
kurodafolio.com                      … ポートフォリオ本体（営業の顔・据え置き）
tools.kurodafolio.com/               … ツールポータル トップ（ツール一覧）  ← 新・層2
tools.kurodafolio.com/image-compress/ … 第1弾「画像仕分け圧縮くん」
tools.kurodafolio.com/clamp/          … clampメーカー（第2弾候補）
tools.kurodafolio.com/color/          … 配色＆コントラスト（候補）
tools.kurodafolio.com/flocss/         … FLOCSSスキャフォルダ（候補）
tools.kurodafolio.com/motion/         … スクロール演出（温存中／旧 motion.kurodafolio.com 案を吸収）
```

- ツール群は **`tools.` サブドメインを1個だけ立て、その配下を全てサブディレクトリ**で集約。
- ★要確認（軽微）: 「ツール群を `tools.kurodafolio.com` サブドメインにするか、本体直下 `kurodafolio.com/tools/` サブディレクトリにするか」。
  - `tools.` 推奨理由: 本体=ポートフォリオ（営業）とツール集の**性格を分離**でき、ツール集を独立ブランド（destination）として育てやすい。本体の被リンク文脈（制作実績）とツールの被リンク文脈（便利ツール）は別なので、混ぜない方が両方の意図が明確。
  - 代替 `kurodafolio.com/tools/` の利点: 本体ドメインパワーを最初から完全継承（サブドメインのゼロスタートすら無い）。本体の評価が十分育っているなら、こちらの方がツール集の立ち上がりは速い。
  - → **本体ポートフォリオの現在のドメインパワー次第**。まだ弱いなら `kurodafolio.com/tools/`（本体継承）、本体を営業の顔として性格を守りたいなら `tools.`。**この1点だけ黒田さんの優先度（立ち上がり速度 vs 性格分離）を確認したい。** Mado推奨は性格分離を取って `tools.`。

### 5-3. 第1弾「画像仕分け圧縮くん」の置き場所

**ポータルの最初の1ツールとして公開する（単独先行しない）。**
- ローンチ時点ではトップのツール一覧に**カードが1枚だけ**並ぶ状態でOK。器（ポータル）を最初から建て、その1区画目に画像圧縮を置く。
- これにより、第2弾clamp・第3弾配色を出すとき「新ツールのカードを足す」だけで済み、**最初から評価が `tools.` ドメインに集約され始める**。単独サブドメインで先行させてしまうと、後でポータルに移すURL引っ越し（被リンク毀損・301運用）が発生するので、**最初からポータルに置くのが将来コストを最小化する**。
- 第1弾は `tools.kurodafolio.com/image-compress/`。トップ `tools.kurodafolio.com/` は「黒田さんのフロントツール集」の顔として同時に用意する。

### 5-4. 「ツールが増える将来」への接続（1段落）

このポータルルートは「自分用フロント開発ツールが複数増えていく」前提（`mado-selfuse-dev-tools.md`）に最も素直に接続する。clamp・配色・FLOCSS・メタ工房・スクロール演出…と増えるたびに、新サブドメインを立てるのではなく `tools.kurodafolio.com/` 配下にページを1枚足す。各ページは自分の検索意図で個別最適され（カバレッジが意図の数だけ広がる）、被リンク・回遊・ドメイン評価は全て `tools.` 1拠点に複利で蓄積される。ブログ記事（FLOCSS/clamp/画像最適化）とツール集が相互送客し、「フロント制作の道具と知識はここに揃う」という destination ブランドが1ドメインに育つ。スクロールツールも `motion.` 独立サブドメイン案を破棄して `/motion/` 配下に吸収すれば、ツール群が綺麗に1拠点へ統一される。運用は1コードベース・1ビルドのまま、ツールの数が増えるほどSEO資産が強くなる——個人事業の保守負荷を上げずにドメインを育てられる、最も損の少ないルート。

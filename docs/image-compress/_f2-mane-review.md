## 判定

**条件付き合格**

設計は案②フル（D1〜D10）に正確に対応し、勝手な拡大・縮小はない。工数の山場4点（複数WASM組込み・種別⇄パラメータマッピング・Web Worker逐次・劣化確認UI）はすべて仕様に対応物を持ち、重点リスク（AVIF固まり・自動推測精度・大量画像メモリ）も対策込みで落ちている。**F3 実装着手は可能**。ただし、F3 で Cody が判断に詰まりうる「実装着手前に潰すべき小さな未確定点」が数点残るため、それらを Must / Should として下記に挙げ、条件付きとする（差し戻し＝再設計ではなく、起動前の補足整理で解消できるレベル）。

## 2. 工数整合（PLAN §3「実働2〜3週・山場」との照合）

PLAN §3 の山場4点が、いずれも仕様に**対応物**を持つ。仕様の作り込み量は案②フルの「2〜3週」見積りと矛盾しない（過大・過小いずれもなし）。

| 山場（PLAN §3） | 仕様の対応物 | 工数整合の所見 |
|---|---|---|
| 複数WASMエンコーダ組込み（3〜4本） | §3 表（webp/avif/oxipng/imagequant）＋ §3-a 遅延import方針 | ✅ エンコーダ4本＋遅延import設計まで具体化。§3 の「3〜4本」と一致。**遅延import は工数を増やす方向だが、初回ロード体験のため正当**（過剰ではない） |
| 種別⇄パラメータマッピング（6種別） | §1 確定表（具体値入り）＋ §1-1 運用ルール | ✅ 6種別×フォーマット/品質/リサイズ/付加処理を**実装者が使える具体値**で確定。フォーマット変更時の品質追従ルール（§1-1）まで踏み込んでおり、マッピング工数を正しく織り込んでいる |
| Web Worker化（AVIF固まり対策） | §3-b（逐次・Transferable・try/catch継続） | ✅ 単なる「Workerでやる」でなく逐次・転送・1枚失敗継続まで設計。§3 の「ブラウザが固まる懸念のケア」に正対 |
| 劣化確認UI | §4-3（並置必須・ルーペ任意・スライダー将来） | ✅ D6 の優先度で工数を段階化。**ルーペを「工数を見て載せる」任意扱い**にしており、工数膨張の安全弁が効いている |

加えて、自動推測（§5）を「**実装は最後に積む**」と明記している点は、工数管理として優秀。手動割当（痛点解消の本丸）が先に動けば、推測が間に合わなくても v1 は成立する設計＝**工数オーバー時の切り離し線が引けている**。

**工数を超えて膨らんでいる箇所**: なし（むしろ任意機能に降格ラインを敷いている）。
**過小評価の箇所**: なし。ただし §5-c（下記 Should S-1）で imagequant の色数サンプリングは「軽量」と書かれているが、減色品質チューニングは実画像での試行錯誤が要る領域＝§3 の「2〜3週」内に収めるには**減色は④のPNG経路に限定し深追いしない**運用を F3 で守る必要がある（仕様には限定の意図はあるが、Cody への明示が望ましい）。

## 3. 実現可能性（ブラウザ内WASM完結の土俵で実装可能な粒度・順序か）

- **6種別⇄パラメータ表**: 具体値・付加処理・PNG可逆経路の品質スライダー無効化（§1-1）まで書けており、Cody が推測せず実装できる粒度。✅
- **WASM選定**: `@jsquash` 系で基軸を確定、遅延import の発火タイミング（そのフォーマット/種別を初めて使う瞬間）も明記。✅
- **自動推測**: §5 の特徴量（解像度・透過・色数・入力形式）はデコード時に軽量取得できるものに限定、②を機械推測から除外する判断（誤推測＝信頼毀損回避）も妥当。✅ ただし「色数サンプリング」の実装手段は未明記（→ Should S-1）。
- **Web Worker 逐次（AVIF固まり対策）**: 逐次＋Transferable＋進捗＋失敗継続で、ブラウザ内完結の土俵で実装可能な順序。✅
- **実装順序**: §2 D2／§5 で「手動割当が動いてから推測を載せる」と順序を指定済み。F3 の着手順（webp基軸→種別マッピング→Worker→劣化UI→推測）が読み取れる。✅

**F3 で Cody が詰まりうる未確定点**（実装ブロッカーではないが、起動前に潰すと手戻りが減る）:

- **(a)** WASM `.wasm` バイナリの配信を「Vite asset・`base:"/tools/"` 配下のパス解決に注意」とだけ書いている（§3-a）。F1 で `base:"/tools/"` のパス解決は検証済み（`00_tech-verification.md` §1/③）だが、**Web Worker 内からの WASM フェッチ**は通常の asset 解決と挙動が異なりうる（Worker の `import.meta.url` 起点）。→ Must M-1。
- **(b)** ②写真(重要) を自動推測から除外した結果、**推測ON時に②に該当する画像が③に割り当たる**。仕様意図（②は手動）は正しいが、UI 上で「②は手動で選んでください」と気づかせる導線が §4/§5 に明記されていない。→ Should S-2。

## 5. 抜け漏れ（実装着手前に潰すべき未決の明示と、ブロッカーの有無）

01_spec §末尾に「★要確認（黒田さん/実機調整）」「★要確認（F3実装者）」が明示され、実機調整事項（リサイズ上限px・品質値・枚数/サイズ上限・減色色数・oxipngレベル）が列挙されている。**これらは実機調整前提で初期値が入っており、実装ブロッカーではない**（Cody は初期値で着手し F4 で詰められる）。

実装ブロッカー級の未決の見落としは**なし**。ただし下記 Must/Should は起動前に明示すると手戻りを防げる。

## 指摘一覧（Must / Should / Note）

### Must（F3 起動前に明示・補足すべき）

- **M-1: Web Worker 内からの WASM 配信パス解決**を F3 プロンプトに前提として渡す。§3-a は「`base:"/tools/"` 配下のパス解決に注意」と書くが、メインスレッドの asset 解決と Worker 内解決は別物。F1 検証（`00_tech-verification.md`）はメインスレッドのパス解決が対象のはず＝**Worker＋WASM の組合せは F3 実装初期に AVIF 固まり検証と同時に最優先で確認**するよう、01_spec §3-b またはプロンプトに一文足すこと。これを詰めずに進むと、AVIF/oxipng の遅延import が Worker 内で解決できず手戻りになりうる。

### Should（推奨・品質を上げる）

- **S-1: 色数サンプリングの実装手段を一言補足**（§5-1）。「縮小サムネ上のユニーク色をサンプリング」とあるが、`OffscreenCanvas` で縮小→`getImageData`→ユニーク色カウント、等の手段を Note レベルで添えると Cody の推測がゼロになる。減色（imagequant）は④PNG経路に限定し深追いしない運用も併記推奨（§2 工数所見と連動）。
- **S-2: 「②は手動選択」導線の明記**（§4-2／§5）。②を推測から除外した帰結として、推測ON時に重要写真が③に落ちる。UI で「重要な写真は②を手動で選んでください」と気づかせる文言/導線を §4 に一行加えると、痛点②（顔写真を潰さない）の取りこぼしを防げる。
- **S-3: Safari の AVIF エンコード可否のフォールバック挙動**を spec 側にも一言。02_qa 2-2 で「不可なら WebP フォールバック」と書かれているが、01_spec 側に「Safari で AVIF エンコード不可時は WebP に自動降格＋通知」の挙動仕様がない。F3 実装の分岐になるため spec に書くのが望ましい（QA で初めて気づくと手戻り）。

### Note（参考・実装裁量）

- **N-1**: §4-1 のフォーマット別容量比較（WebP vs AVIF vs 元）は「任意表示」だが、AVIF を実エンコードしないと比較値が出ない＝**比較表示は計算コストを伴う**。任意機能として工数の安全弁内にあるが、F3 で「ボタン明示時のみ AVIF を試算」とする等、無条件全枚 AVIF 試算を避ける運用が望ましい（メモリ・速度の観点）。
- **N-2**: ZIP の `fflate`「超大量は分割DL案内」の閾値が未定。D8 の枚数上限（20〜30枚）内なら通常 ZIP で足りるはずで、実機調整事項に含めてよい。

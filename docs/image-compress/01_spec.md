## 0. 設計思想（痛点への回答の再定義・2026-06-17 確定）

痛点「TinyPNG は圧縮率も方式も選べない」への回答は、**2軸の選択**で担保する。**数値スライダー（品質値・長辺px）は外す**。

| 軸 | ユーザー操作 | 意味 |
|---|---|---|
| **① 種別を選ぶ** | 6種別のラジオボタン（一括＋個別） | 圧縮の**意図**を選ぶ（背景は強く／重要写真は画質優先／…）。品質値・リサイズ上限は種別ごとに内部固定で自動適用（§1-1） |
| **② 出力フォーマットを選ぶ** | チェックボックス（複数選択可） | WebP / AVIF / PNG可逆 / 元形式（JPEG）から欲しい形式を**複数**選べる。選んだ分だけ出力して ZIP 同梱（§4-1-4） |

- **「選べる自由＋おまかせの両立」は、数値スライダーではなく『種別＋フォーマット選択』で実現する**——これが本ツールの再定義。黒田さんの実体験＝「品質数値・長辺pxは定量的に分かりづらく、種別を選べば適切な値が当たる方が良い」に基づく。
- これにより**オーバーライドUIは大幅に簡素化**され、ユーザーが触る操作面は「種別ラジオ ＋ フォーマットのチェックボックス」だけになる（quality/resize の数値展開UIは消える）。
- §1 表の「品質値」「リサイズ上限」列は**残す**（実装の内部固定値・実機調整対象）。ただし**UIには出さず・種別選択で自動適用**する（§1-1）。

## 1. 種別プリセット（6種別）⇄ パラメータ確定表

各プリセットは**種別ごとに固定された圧縮方針**。品質値・リサイズ上限px は**エンジン内部の固定値**であり、**UIには出さない**（黒田さん確定 2026-06-17・後述§1-1）。ユーザーは「種別を選ぶ＝圧縮の意図を選ぶ」だけでよく、品質数値・長辺pxの微調整はしない。品質値は各 WASM エンコーダの quality（0–100）を指す。リサイズは**長辺（width/height の大きい方）の上限px**で、上限以下の画像は拡大しない（縮小のみ・アスペクト比維持）。**ユーザーが変更できるのは「種別」と「出力フォーマット（複数選択可）」の2軸のみ**（§4-1・§4-1-4）。

| # | 種別 | 想定画像 | 推奨フォーマット（既定チェック） | 品質値（内部固定・UI非表示） | リサイズ上限（長辺・内部固定） | 付加処理 | 設計意図 |
|---|---|---|---|---|---|---|---|
| ① | **背景・装飾＝強圧縮** | ヒーロー背景・セクション背景・テクスチャ | WebP（非可逆） | **WebP q=55**（AVIF出力時 q=45） | **1920px** | EXIF削除・Orientation正立 | 視認の主役でないため容量最優先。多少のノイズ許容。黒田さんの「背景は強く潰す」主痛点①に直結 |
| ② | **写真(重要)＝画質優先** | 顔写真・人物・商品 | WebP（非可逆） | **WebP q=85**（AVIF出力時 q=63） | **2560px**（過度にリサイズしない） | EXIF削除・Orientation正立 | 劣化が信頼を損なう領域。シャープネス維持・強い縮小をしない。主痛点②「顔写真を潰さない」に直結 |
| ③ | **写真(一般)＝中圧縮** | 記事中の説明写真・風景 | WebP（非可逆） | **WebP q=72**（AVIF出力時 q=52） | **1600px** | EXIF削除・Orientation正立 | 主役でも装飾でもない中間。容量と画質のバランス。主痛点③に直結 |
| ④ | **イラスト・図版** | フラットイラスト・図解・グラフ | WebP（非可逆）＋PNG可逆（減色） | **WebP q=80**／PNG経路は**減色 256色**（実機調整・§3-d） | **1600px** | EXIF削除・**imagequant 量子化**（PNG経路時） | 平坦色は減色がよく効く。JPEG系のモスキートノイズを避けWebP/PNG寄りに。グラデ有りはWebP中圧縮 |
| ⑤ | **アイコン・ロゴ（SVG助言）** | UIアイコン・ロゴ・線画 | **PNG可逆（oxipng）** | **可逆**（品質劣化なし・oxipng最適化レベル内部固定 **2**） | **リサイズしない**（透過保持・元寸維持） | EXIF削除・透過保持・**「SVGにできませんか?」助言バナー表示** | 小さく輪郭が命。非可逆で輪郭が滲むと致命的。ラスタ最適化はするが、根本解は SVG 化という助言を必ず出す（D5・文字列のみ＝低コスト） |
| ⑥ | **スクショ・UI（可逆優先）** | 画面キャプチャ・管理画面 | **PNG可逆（oxipng）** | **可逆**（文字が滲まない・oxipng最適化レベル内部固定 **2**） | **リサイズしない**（実機調整・等倍書き出し前提） | EXIF削除・**Orientation は通常不要だが念のため正立** | 文字・線の可読性が命。非可逆だと文字が滲む。黒田さんは記事でスクショ多用＝可逆を既定に |

### 1-1. 表の運用ルール（実装者向け）

- **品質値・リサイズ上限px は「種別選択で自動適用される内部固定値」であり、UIには出さない**（黒田さん確定 2026-06-17）。品質スライダー（quality / oxipngレベル）UI・長辺px入力UI は**実装しない**。種別を選べば適切な値が自動で当たる。これは黒田さんの実体験＝「数値が定量的に分かりづらく、種別を選べば適切な値が当たる方が良い」に基づく確定。
- **大きすぎる画像は種別ごとの上限px へ自動縮小**する（縮小のみ・アスペクト比維持）。ユーザーには「Web表示に十分なサイズへ自動縮小」と一言伝わるようにする（§4-1-3 のヒント or 種別説明文の近傍）。
- **「推奨フォーマット（既定チェック）」は初期チェック状態**。ユーザーは出力フォーマットをチェックボックスで複数選択でき（§4-1-4）、チェックした分だけ出力して ZIP に同梱する。
- **フォーマット別の品質値は種別×フォーマットで内部固定**。同じ種別で WebP と AVIF を両方チェックした場合、WebPは表のWebP値・AVIFは表のAVIF値で各々エンコードする（②なら WebP q=85 / AVIF q=63）。**AVIF の品質値は WebP より低い数値で同等の見た目**になる（実機調整）。
- **PNG可逆（⑤⑥・④のPNG経路）は quality 概念がない**（可逆）。oxipng 最適化レベルは内部固定（既定2・UI非表示）。④の減色（256色）も内部処理でUIに数値を出さない。
- **リサイズ上限px・品質値は実機調整**。①1920 / ②2560 / ③1600 等は黒田さんの実案件の書き出しサイズで詰める（★要確認・実機調整事項）。**ただし調整対象はあくまで内部固定値であり、UIに昇格させない**。

### 1-2. 種別の説明文（対外UI文言・確定）

> 各種別ラジオボタンのラベル直下に、以下の説明文を1行で表示する（§4-1-2 / §7-1 言語不問UIのアイコン＋短い説明と整合）。フロント制作者でなくても何を指すか分かる平易な文言。**専門用語を避け、具体例で示す**。

| # | 種別ラベル | 説明文（画面表示・確定） |
|---|---|---|
| ① | 背景・装飾 | ページの背景やテクスチャなど、主役ではない大きな画像。容量を最優先で強く圧縮します。 |
| ② | 写真（重要） | 顔写真・人物・商品など、きれいに見せたい写真。画質を優先して圧縮します。 |
| ③ | 写真（一般） | 記事中の説明写真や風景など、ふつうの写真。画質と容量のバランスをとります。 |
| ④ | イラスト・図版 | フラットな塗りの絵・図解・グラフ。色数が少なく輪郭がはっきりした画像。 |
| ⑤ | アイコン・ロゴ | UIアイコン・ロゴ・線画など、小さくて輪郭が大切な画像。にじませず劣化なしで圧縮します。 |
| ⑥ | スクショ・UI | 画面キャプチャや管理画面など、文字や線がはっきり写った画像。文字をにじませず劣化なしで圧縮します。 |

> 説明文に「自動縮小」を直接織り込まず、§4-1-3 で別途「Web表示に十分なサイズへ自動で縮小します」と一度伝える（種別ごとに繰り返さない）。

## 3. WASM エンコーダ選定

技術土俵＝ブラウザ内 WASM 完結・サーバー不要・画像を送信しない。`@jsquash` 系を基軸とする。

| ライブラリ | 役割 | 初期化コスト | 方針 |
|---|---|---|---|
| **`@jsquash/webp`** | WebP エンコード（可逆/非可逆）・全種別の基軸 | 中 | 必ずロード（最頻出フォーマット） |
| **`@jsquash/avif`** | AVIF エンコード（次世代・容量最小） | **大（重い）** | **遅延 import**（AVIF を初めて選択した時にロード）。Web Worker 逐次必須（§3-b） |
| **`@jsquash/oxipng`** | PNG 可逆最適化（⑤⑥・④PNG経路） | 中 | PNG可逆経路で使用。遅延 import |
| **imagequant 系 WASM**（`@jsquash/png` の量子化 or `imagequant`） | 減色・量子化（④イラスト） | 中 | ④の減色経路でのみ使用。遅延 import |
| **Canvas / `createImageBitmap` / `OffscreenCanvas`** | デコード・リサイズ・Orientation正立・JPEG穏やか最適化（D7） | 小（ネイティブ） | リサイズと D7 のJPEG再エンコードはネイティブで賄う（WASM不要） |
| **`fflate`**（軽量ZIP） | 一括 ZIP 生成 | 小 | DL時のみ。ストリーミング寄り。超大量は分割DL案内 |

### 3-a. 遅延 import 方針（初期化コスト対策）

- 初回ロードは `@jsquash/webp` ＋ Canvas 系のみ（最小バンドルで即操作可能に）。
- **AVIF / oxipng / imagequant はユーザーがそのフォーマット・種別を初めて使う瞬間に dynamic import** する。使わないユーザーに重い WASM をダウンロードさせない。
- WASM の `.wasm` バイナリは Vite の asset として配信（`base:"/tools/"` 配下のパス解決に注意＝F1検証済みの土俵に乗せる）。
- **★F2評価反映（Must M-1）＝Web Worker 内からの WASM パス解決を実装初期に最優先確認**: F1 で検証済みの `base:"/tools/"` のパス解決は**メインスレッド**の asset 解決が対象。エンコードは Web Worker で行う（§3-b）ため、**Worker 内からの `.wasm` フェッチは通常の asset 解決と挙動が異なりうる**（Worker の `import.meta.url` 起点でパスが解決されるため）。**実装初期に、AVIF固まり検証（§3-b・下記★要確認）と同時に「Web Worker 内から AVIF/oxipng/imagequant の遅延 import と `.wasm` 取得が正しく解決できるか」を最優先で確認すること**。これを詰めずに進めると、AVIF/oxipng の遅延 import が Worker 内で解決できず、Worker 構成そのものの作り直し＝大きな手戻りになりうる。

### 3-b. Web Worker 逐次処理（★最重要リスク＝AVIFの固まり対策）

- **エンコードは必ず Web Worker で行う**（メインスレッド凍結＝「壊れた」誤認を防ぐ・非機能の生命線）。
- **逐次処理（1枚ずつ順番に）を基本**とする。並列で複数 WASM を同時に走らせるとメモリが落ちる。AVIF は1枚あたり数百ms〜秒オーダーで特に重いため、**逐次＋進捗バー＋「AVIFは時間がかかります」注記**を必ずセットで出す。
- Worker への受け渡しは `Transferable`（ArrayBuffer / ImageBitmap）でコピーコストを抑える。
- 1枚失敗しても次へ進む（`try/catch` で当該1枚だけエラー表示・全体は止めない）。
- **★F2評価反映（Should S-3）＝AVIF不可環境（Safari等）は WebP へ自動降格**: ブラウザによっては AVIF エンコードが利用不可（Safari 等で `@jsquash/avif` が動作しない／WASM 初期化に失敗する）場合がある。**AVIF を選択した状態でエンコードが不可と判明した場合は、当該フォーマットを WebP に自動降格し、ユーザーに「この環境では AVIF が使えないため WebP で出力しました」と通知する**（処理は止めない）。降格判定は、起動時の AVIF サポート probe（小サイズ画像の試行エンコード）か、初回 AVIF エンコード時の例外捕捉のいずれかで行う。F4 で Safari 実機の挙動を再確認する（02_qa 2-2 と対応）。
  - **★Stage2 実装メモ（Cody / 2026-06-18）**: `@jsquash/avif` の公開 `init()` は内部で `threads()` 判定により **マルチスレッド版（`avif_enc_mt.*` ＋ SharedArrayBuffer ＝ COOP/COEP isolation 必須）** を引きうる。本サイトは isolation 非対応のため、**単一スレッド版ファクトリ `avif_enc.js` を直接 import して自前 instantiate** する方式を採用した（MT 経路に構造的に入らない＝実行時 SAB 不在も、Rollup が MT グルー＋nested worker を変換して起こす **ビルド OOM** も同時に回避）。降格判定は **初回エンコード時の例外捕捉**方式を採用（probe は不採用）。降格通知は実行完了時に1回だけ集約（S-2）。実装詳細は `_f3-stage2-cody-note.md` §1・§6 参照。

### 3-c. 出力フォーマットの段階的有効化（実装範囲の明示・Cody 向け）

出力フォーマットは**チェックボックスで複数選択**できる（§4-1-4）が、**実エンコードの稼働は Stage 単位**。チェックボックスUIと複数出力パイプライン（選んだ分だけエンコードして ZIP 同梱）の骨格は早期に作るが、各フォーマットのエンコードは下表の Stage で有効化する。**未実装段階のチェックボックスは disabled（淡色＋「準備中」表示）でチェック不可**にし、実装範囲を誤認させない。

| 出力フォーマット | エンコーダ | 有効化 Stage |
|---|---|---|
| WebP（可逆/非可逆） | `@jsquash/webp` | **Stage1（稼働済み）** |
| AVIF | `@jsquash/avif`（単一スレッド版 `avif_enc.js` を直接 instantiate） | **Stage2（稼働済み）** |
| PNG可逆（oxipng） | `@jsquash/oxipng` | Stage3 |
| PNG減色（imagequant・④） | imagequant 系 | Stage3 |
| JPEGとして最適化（軽め・D7） | native Canvas | Stage1〜（低コスト・WebPと同時期に載せてよい） |

## 4. UI 設計（共通枠＝`design-concept.md` の世界観に乗せる）

> ツール本体エリア（`--tool-panel-*` の器）の中に以下を配置する。ヘッダー/フッター/パンくず/制作者カード/CTA は共通枠（partial）が担うため本書では扱わない。操作面は `design-concept.md` §2-2 の差分トークン（`--tool-panel-*` / `--dropzone-*` / `--field-*`）を使う。

> **★UI再設計（黒田さん確定 2026-06-17・実機確認後）**: 種別選択は**ラジオカード**（プルダウン廃止）。出力フォーマットは**チェックボックス（複数選択可）**。**品質スライダー・長辺px入力UIは削除**（種別ごとに内部固定値を自動適用＝§1-1）。ユーザーが触る操作面は「①種別ラジオ ＋ ②出力フォーマットのチェックボックス」の2軸のみに簡素化される。
>
> **★確定形（黒田さん実機FB確定 2026-06-18・実装が正）**:
> - **種別・出力フォーマットとも「一括＋カード個別」の二段構え＝本実装で確定**（前回まで行別フォーマット上書きは「任意・工数次第」だったが、黒田さん要望で本実装＝確定）。各行カードの「種別を選択」「出力形式」トグルで個別の種別ラジオ群・フォーマットチェックボックス群を開閉できる。
> - **各行カードに選択中の種別・形式をテキストで常時表示**（例:「種別: ②写真（重要） ／ 形式: WebP・JPEG」）。種別・形式を変えると追従更新する。
> - **「種別を選択」「出力形式」トグルは三角アイコン（▶閉／▼開）で開閉状態を示す**（開状態のテキストは accent 色）。
> - **after画像の並置は廃止**（§4-3）。劣化確認は容量・削減率の数値で行う。
> - **②昇格導線（promote / 旧 S-2）は撤去**（§4-2）。②は手動選択前提を維持するが画面上の昇格ヒントは出さない。

### 4-1. 画面フロー

1. **投入（ドロップゾーン）**: 画面上部に破線ボーダーのドロップゾーン（`--dropzone-*`）。D&D ＋「ファイルを選択」ボタン（D&D代替・A11y/モバイル必須）。ドラッグオーバーで accent 破線＋accent背景。フォルダ一括は任意（`webkitdirectory`）。
2. **全体一括の種別選択（ラジオボタン）**: 投入直後、画面上部に「全部の種別を一括で：」＋**6種別のラジオボタン群**（①〜⑥／各ラベル直下に §1-2 の説明文を表示）＋「自動で種別を推測 ON/OFF」トグル（D2）。ラジオ＝1つだけ選べる＝一括で当ててから個別に直す動線。
3. **画像リスト（1枚=1行カード）**: 各画像に（投入画像の）サムネ＋ファイル名＋**選択中の種別・形式のテキスト常時表示**（例:「種別: ②写真（重要） ／ 形式: WebP・JPEG」）＋「種別を選択」「出力形式」トグル（三角▶▼開閉・開状態テキストは accent 色／開くと個別の種別ラジオ群・フォーマットチェックボックス群）＋ **圧縮前後の容量・削減率の数値表示**（劣化確認＝§4-3・after画像の並置は行わない）。**画面のどこか（リスト上部 or 各行のヒント）に「大きすぎる画像は Web 表示に十分なサイズへ自動で縮小します」と一度だけ表示**（§1-1・自動縮小の告知）。
4. **出力フォーマット選択（チェックボックス・複数選択可）**: 種別ごとに**推奨フォーマットが既定でチェック**された状態（§1 表「推奨フォーマット」列）。ユーザーは**チェックした分だけ出力**でき、ZIP に同梱される（例:「PNGを圧縮して WebP と 圧縮PNG の両方を出力」）。選択肢＝WebP ／ AVIF ／ PNG可逆（oxipng）／元形式（JPEG・D7）。フォーマット別容量比較（WebP vs AVIF vs 元）を任意表示（複数チェック時に価値が出る）。**品質スライダー・oxipngレベル・リサイズ上限px の入力UIは出さない**（§1-1・内部固定）。
   - **配置**: 一括設定エリア（§4-1-2 近傍）で「全体の既定フォーマット」を、各行カードで「その画像だけ別フォーマット」を上書きできる二段構え（種別と同じ一括＋個別パターン）。**行別上書きは本実装＝確定**（黒田さん要望 2026-06-18／前回「任意」から確定へ昇格）。各行カードの「出力形式」トグルでチェックボックス群を開閉する。
   - **段階的有効化（実装範囲の明示）**: チェックボックスUIと複数出力パイプラインは作るが、**実エンコードは段階稼働**＝WebP は Stage1 稼働済み／AVIF は Stage2／PNG可逆(oxipng)・PNG減色(imagequant) は Stage3。**未実装段階のフォーマットは disabled（淡色＋「準備中」等）で示し、チェック不可にする**（§3-c も参照・Cody が実装範囲を誤認しないように）。
5. **処理・進捗**: 「圧縮する」実行で Web Worker 逐次処理。進捗バー（n/N 枚）＋現在処理中ファイル名。AVIF含む時は「時間がかかります」注記。
6. **出力**: 各行に個別DL＋ 全体「一括ZIPダウンロード」。**複数フォーマットを選んだ場合は選んだ数だけファイルが出力され**、ZIP に同梱。出力ファイル名＝**元名維持＋各拡張子追従**（`photo.png` → `photo.webp` ＋ `photo.png`（圧縮PNG）等／同名でも拡張子違いで衝突しない・D10）。

### 4-2. 種別の割り当て方（UX の肝）

- **一括**（全画像に同じ種別・ラジオ1択）＋**個別**（各行のラジオで1枚ずつ変更）の二段構え。
- **自動種別推測 ON 時**は各画像にデフォルト種別が初期選択される（§5）。OFF 時はユーザーが一括/個別で割り当てる。
- 痛点の本質は「選べない」こと＝**手動の種別選択＋出力フォーマット選択（複数可）が最優先機能**。自動推測ゼロでも痛点は解消する設計にする（推測は付加価値）。
- **S-2（②昇格導線）は実装後に黒田さん判断で撤去（2026-06-18）**: F2評価では「②写真(重要)へ手動で上げる導線を明示」を Should S-2 として提案していたが、黒田さんの実機確認後の判断で**画面上の昇格ヒント／導線は撤去**した。②はユーザー手動選択前提を維持する（②は機械推測の対象外＝§5）が、③割当カード等に「②へ上げてください」と気づかせる文言・ヒントは出さない。②の選択肢自体は種別ラジオに常設されており、ユーザーは一括・個別のラジオで②をいつでも選べる。
  - （参考・撤去理由）昇格ヒントは画面のノイズになり、種別ラジオに②が常設されている以上、手動選択の動線は既に確保されている、という黒田さんの実機判断による。

### 4-3. 劣化確認 UI（D6）

- **圧縮前後の容量・削減率の数値表示（必須）**: 各行カードに圧縮前容量 → 圧縮後容量＋削減率（例: `1.2MB → 340KB / -72%`）を表示し、劣化の度合い・効果を数値で確認する。複数フォーマット出力時はフォーマット別に容量・削減率を併記。
- **after画像の並置は行わない（黒田さん確定 2026-06-18）**: F2 設計時点（D6）では「Before/After サムネ並置＝必須」としていたが、黒田さんの実機確認後の判断で**after サムネ画像の並置は廃止**した。劣化確認は上記の容量・削減率の数値で行う。before サムネ（投入画像のサムネ）はカード左に表示するが、after の縮小画像は生成・表示しない。
- **ルーペ（廃止）**: after 並置を前提とした拡大比較機能のため、並置廃止に伴い v1 では作らない。
- **スライダー重ね比較（将来）**: v1 では作らない。

### 4-4. ⑤アイコン種別の SVG 助言（D5）

⑤を割り当てた画像には「**このアイコンは SVG にできませんか? SVG なら拡大しても劣化せず容量も小さくできます**」の助言バナーを表示（文字列のみ・低コスト・差別化）。SVG 自体の最適化（SVGO相当）は v1 では行わない。

## 5. 自動種別推測ロジック（D2）

> ON/OFFトグル（初期値=ON）。推測は**初期値**であり、必ず手動で直せる。推測根拠を各画像に小さく表示。**精度は追わない（当たればラッキー設計）。実装は最後に積む**（手動割当が完成してから載せる）。

### 5-1. 判定に使う特徴量

デコード時に取得できる軽量な特徴量のみを使う（重い解析はしない）:

- **解像度（長辺px・アスペクト比）**
- **透過アルファの有無**（PNG/WebP のアルファチャンネル）
- **色数（おおよそ）**: 縮小サムネ上のユニーク色をサンプリングして「少色（〜数百色）か多色か」を判定
  - **★F2評価反映（Should S-1）＝実装手段**: `OffscreenCanvas`（または通常 Canvas）に**縮小サムネ（例: 長辺 64〜128px）を描画 → `getImageData` で画素配列を取得 → ユニーク色を Set でカウント**（厳密な全色カウントではなく上限到達で打ち切る軽量判定で十分。例: ユニーク色が数百を超えた時点で「多色」と確定し走査を止める）。フル解像度では走査しない（縮小サムネ上でのみ判定＝軽量）。なお、この色数判定は**自動推測（種別割当）用の軽量サンプリング**であり、④イラストの**減色（imagequant による量子化）とは別物**。減色品質チューニングは実画像での試行錯誤を要するため**④の PNG 経路に限定し、深追いしない**（工数の安全弁・§2 工数所見と連動）。
- **入力フォーマット**（PNG/JPEG/WebP）

### 5-2. デフォルト割当ヒューリスティック（目安・実機調整）

| 条件（上から優先） | 推測種別 |
|---|---|
| 透過あり ＆ 長辺が小さい（〜512px目安）＆ 少色 | ⑤ アイコン・ロゴ |
| 透過あり ＆ 少色（フラット） | ④ イラスト・図版 |
| 入力が PNG ＆ 多色 ＆ アスペクト比が画面的（16:10〜16:9近辺） | ⑥ スクショ・UI |
| 入力が JPEG ＆ 横長大サイズ（長辺 1600px超）＆ アスペクト比が広い | ① 背景・装飾 |
| 入力が JPEG（一般的な写真サイズ） | ③ 写真(一般) |
| 上記いずれにも当てはまらない | ③ 写真(一般)（無難なフォールバック） |

> ②写真(重要)は機械的に推測しない（「重要かどうか」は機械判定不能＝誤推測が信頼毀損になる）。②はユーザーが手動で選ぶ前提とし、推測は①③④⑤⑥の範囲に留める。各画像に「（推測: 透過＋小サイズ→アイコン）」のように根拠を small で表示する。

## 6. 入力・出力仕様

### 6-1. 入力

| 項目 | 仕様 |

## 7. ★i18n 代替＝言語不問 UI（U-05 / U-06）

> 日英切替（i18n）は**採用しない**（U-05確定）。代わりに**言語に依存しないUI**を設計の前提に織り込む。

### 7-1. 言語不問 UI の設計原則

- **アイコン＋数値主導**: 操作の主役（圧縮実行・DL・種別・容量・削減率）は**アイコン＋数値**で伝わるようにする。容量（`1.2MB → 340KB`）・削減率（`-72%`）・進捗（`n/N`）は言語非依存で読める。（※品質スライダーは廃止＝§1-1。品質値はUIに出さない＝言語非依存性に影響しない。）
- **種別はアイコン＋ラジオ＋短い説明で識別**: 6種別ラジオはラベル文字だけに頼らず、各種別に識別アイコン（背景＝山/写真＝カメラ/イラスト＝絵筆/アイコン＝星/スクショ＝モニタ 等のLucideライクなSVG）＋ §1-2 の短い説明文を添え、ラベルが読めなくても種別が区別できるようにする（アイコン＋短い説明＝§7 の言語不問UIと整合）。
- **出力フォーマットはチェックボックス＋拡張子表記**: 複数選択のフォーマットはアイコン＋拡張子文字（`WebP` `AVIF` `PNG` `JPEG`）で示し、説明文に頼らず識別できるようにする。準備中（未実装段階）は disabled の見た目で示す（§3-c）。
- **劣化確認は数値で伝える**: 圧縮前後の容量・削減率（`1.2MB → 340KB / -72%`）は言語非依存で読める。after画像の並置は行わない（§4-3）。
- **エラーはアイコン＋最小語**: スキップ/非対応は ⚠ アイコン＋短い語で示し、長文説明に頼らない。

### 7-2. 将来余地（U-06 復帰条件成立時）

- **プリセット名の英語併記**の余地を残す（例: `背景・装飾 / Background`）。マークアップ上、種別ラベルは1テキストノードに閉じ込めず将来併記しやすい構造にしておく（ただし v1 では日本語のみ）。
- U-06 の発火条件（海外実需・ドメイン成熟・複数人運営のいずれか）成立時に、本 §7 の延長で日英切替を再評価する。

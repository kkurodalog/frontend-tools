## 1. ツールの目的・対象・スコープ

### 1-1. 解決する課題（再定義）
TinyPNG型の「全部おまかせ・圧縮率も方式も選べない」一律圧縮では、**背景画像（容量最優先で強く潰したい）と顔写真（劣化させたくない）を同じ強さで処理してしまう**。本ツールは**画像をその役割（種別）で仕分けし、種別ごとに最適な圧縮方針を当て、かつ手動で選び直せる**ことで、この痛点をピンポイントで解消する。あわせて**ブラウザ内WASM完結＝画像を送信しない**ことで、TinyPNGのアップロード型に対するプライバシー差別化を実現する。

### 1-2. 対象ユーザー
| 区分 | 誰 | ニーズの中心 |
|---|---|---|
| **主** | 黒田さん本人（毎案件・毎ページの画像書き出し直後に使用） | 種別ごとに圧縮を選び分けたい・自分のワークフローで毎回使える速さ |
| **従** | 同業フロントエンジニア／ブロガー（公開後流入） | 「背景と顔写真を一律で潰したくない」潜在不満・プライバシー（送信しない）・無料無制限 |

> 主＝自分用のドッグフーディングが選定動機。従＝公開時の刺さりは副次。**主のために設計し、従に配慮する**順序を崩さない（迷ったら主の作業効率を優先）。

### 1-3. スコープ宣言（何をやる／やらないツールか）
- **やること**: ブラウザ内で完結する「画像最適化（圧縮・フォーマット変換・必要なリサイズ）」。種別プリセットによる方針自動当て＋手動オーバーライド＋Before/After確認＋一括DL。
- **やらないこと（§2-5・YAGNI排除）**: 画像編集（トリミング/フィルタ/補正/加工）、画像生成AI、アップロード保存・クラウド同期、アカウント/ログイン、URL逆引き、案件管理。**「最適化に集中し、編集も保存もしない」**を一線とする。

## 2. 機能要素の棚卸し（必須v1 / 任意v1 / 将来 / 不要）← 本書の核心

判定凡例: **【必須v1】**=フル品質に不可欠 / **【任意v1】**=工数次第で入れる / **【将来】**=v2以降 / **【不要】**=やらない（YAGNI・スコープ外）

### 2-1. 入力
| 要素 | 判定 | 理由 |
|---|---|---|
| D&D（ドロップ） | **必須v1** | 毎回の起点。最速の投入動線 |
| ファイル選択ボタン | **必須v1** | D&D代替（A11y・モバイル）。低コスト |
| 複数枚一括投入 | **必須v1** | 「毎ページ複数画像をまとめて」が実使用の前提 |
| 入力: PNG | **必須v1** | 制作の主力（透過・図版・スクショ） |
| 入力: JPEG | **必須v1** | 写真の主力 |
| 入力: WebP | **必須v1** | 再圧縮・既存資産の変換需要。デコードは標準API/WASMで軽い |
| 入力: GIF | **任意v1** | 静止GIFは受けてもよいが需要薄。**アニメGIFは非対応明記**（動画最適化は別領域＝スコープ外） |
| 入力: AVIF | **任意v1** | 入力で受ける需要は薄い。デコードは可能だが優先度低 |
| 入力: **HEIC**（iPhone写真） | **【将来】**（v1は不可・明示） | iPhone実機写真の最適化ニーズはあるが、**HEICはブラウザネイティブデコード不可**。libheif-wasm等の追加が必要で工数・バンドルが跳ねる。「黒田さんの制作素材＝デザイン書き出し（PNG/JPEG）」が主なので**v1痛点には不要**。受けるなら「HEIC→事前にJPEG化してから」案内で代替。★DECISION |
| 入力: SVG | **【将来 or 助言のみ】** | SVGの「最適化」（SVGO相当）は別機能群。v1では**アイコン種別で「SVGにできませんか?」助言を出す**に留め、SVG自体の最適化は載せない |
| フォルダごと一括（ディレクトリ投入） | **任意v1** | `webkitdirectory`で実現可だが、複数枚D&Dで実用上足りる。優先度中 |
| クリップボード貼付（Ctrl+V） | **【不要】** | スクショ1枚を直貼りする以外に使い所が薄い。複数一括の本ツールと相性弱。YAGNI |
| 入力枚数上限 | **必須v1**（緩い上限＋警告） | メモリ/フリーズ対策の実務必須。無制限を謳わず「一度に〜N枚目安」のガード＋逐次処理。★DECISION（N値） |
| 入力ファイルサイズ上限 | **必須v1**（1枚あたりの上限警告） | 巨大画像でWASMがメモリ落ちする事故防止。超過時はスキップ＋メッセージ |

### 2-2. 種別プリセット（§E-1の6種別を再検討）
| 要素 | 判定 | 理由 |
|---|---|---|
| ①背景・装飾＝強圧縮 | **必須v1** | 本人主痛点①。プリセットの中核 |
| ②写真（重要）＝画質優先 | **必須v1** | 本人主痛点②（顔写真を潰さない）。中核 |
| ③写真（一般）＝中圧縮 | **必須v1** | 本人主痛点③。中核 |
| ④イラスト・図版（減色/量子化） | **任意v1**（フル決定なら採用） | 平坦色に減色が効くのは正しいが、減色（imagequant）導入で工数増。痛点の核ではない「あれば嬉しい」域 |
| ⑤アイコン・ロゴ（SVG助言＋可逆） | **任意v1**（助言は低コスト・採用推奨） | 「SVGにできませんか?」助言は**文字列を出すだけで低コスト・差別化大**。可逆PNG最適化はoxipng導入次第 |
| ⑥スクショ・UI（可逆優先） | **任意v1** | 文字の滲み回避＝可逆。oxipng導入が前提。黒田さんはスクショを記事で多用するなら価値あり。★DECISION（記事スクショ頻度次第） |
| 自動種別推測（解像度/透過/色数からデフォルト割当） | **【任意v1・要シビア判定】** | 便利だが**推測ミスが必ず出る**（横長でも顔写真はある等）。誤推測は「勝手に潰された」体験＝信頼毀損。入れるなら「**推測は初期値・必ず手動で直せる・推測根拠を小さく表示**」を厳守。精度を追わず「当たればラッキー」設計に留める。★DECISION（v1で入れるか） |
| 手動オーバーライド（種別変更プルダウン） | **必須v1** | 痛点の本質は「選べない」こと。**選べる自由が最優先機能**。自動推測を入れなくてもこれは必須 |
| 手動オーバーライド（quality/フォーマット/リサイズの個別調整） | **必須v1** | プリセットはあくまで初期値。玄人の微調整窓口 |
| プリセットのカスタム保存（自分用プリセット追加） | **【将来】** | localStorageで実現可だが、6種別で実用上足りる。v2の効率化ネタ |

### 2-3. 圧縮 / 変換
| 要素 | 判定 | 理由 |
|---|---|---|
| 出力: WebP（非可逆/可逆） | **必須v1** | 全種別の基軸フォーマット。`@jsquash/webp` |
| 出力: AVIF | **任意v1**（フル決定なら採用 / ★重さ注意） | 次世代・容量最小で公開時の刺さり大。ただし**エンコードが重い**→Web Worker＋逐次＋「重い」注記が前提条件。痛点の核ではない |
| 出力: PNG可逆（oxipng） | **任意v1** | スクショ/アイコンの可逆需要に必要。④⑤⑥を採るなら連動して必要 |
| 出力: 元形式維持 | **任意v1** | 「JPEGのまま軽くしたい」需要。MozJPEG相当が要るなら工数増。WebP変換で足りるなら不要寄り |
| 品質スライダー（quality粒度） | **必須v1** | オーバーライドの中核。粒度は1刻みで十分（0-100スライダー） |
| 可逆圧縮・減色（量子化） | **任意v1** | ④⑤⑥に連動。imagequant系WASM導入。痛点核ではない |
| リサイズ（長辺上限） | **必須v1** | 背景1920px上限等、プリセットに内包。容量削減の主要因 |
| リサイズ（倍率/任意指定） | **任意v1** | 長辺上限で実用は足りる。任意指定は玄人向けで優先度中 |
| メタデータ/EXIF削除 | **必須v1（デフォルトON）** | **容量削減＋プライバシー（位置情報GPS等の混入防止）**。Web書き出し用途でEXIFは基本不要。ただし**Orientation（向き）だけは削除前にピクセルへ焼き込む**処理が必須（後述§3）。★DECISION（デフォルトON/OFF） |
| カラープロファイル(ICC)の扱い | **任意v1（既定: sRGB想定で素通し or 破棄）** | Web用途はsRGB前提。広色域厳密対応はv1不要。**「Web用sRGB前提」と明記**して深追いしない |
| Orientation（EXIF回転）正規化 | **必須v1** | EXIF削除すると回転情報が失われ画像が倒れる事故が起きる。**削除前にCanvasで正立させる**のは必須実装 |

### 2-4. 出力 / UX
| 要素 | 判定 | 理由 |
|---|---|---|
| Before/After 容量・削減率表示 | **必須v1** | ツールの効果可視化＝存在意義。最重要表示 |
| 個別ダウンロード | **必須v1** | 1枚だけ欲しい時の動線 |
| 一括ZIPダウンロード | **必須v1** | 複数枚運用の前提。`fflate`等の軽量ZIP |
| 出力ファイル名規則（元維持 / .webpリネーム / 連番） | **必須v1（元維持＋拡張子変更を基本）** | 拡張子はフォーマット変換に追従必須。連番は任意 |
| 処理進捗表示（枚数/バー） | **必須v1** | 複数枚＋WASMで待ち時間が出る。無反応はフリーズ誤認を生む |
| Web Worker化 | **必須v1** | 大量時のメインスレッド凍結回避。AVIF採用なら特に必須。**非機能の生命線** |
| 劣化確認: Before/After並置 | **必須v1** | 重要写真の画質チェック支援。サムネ2枚並べは低コスト |
| 劣化確認: ルーペ（拡大） | **任意v1** | 並置で足りる場面が多い。ルーペは作り込み度が上がる |
| 劣化確認: スライダー比較（重ねて境界ドラッグ） | **【将来】** | リッチだが工数大。並置/ルーペで代替可。YAGNI寄り。★DECISION |
| フォーマット別容量比較（WebP vs AVIF vs 元） | **任意v1**（AVIF採用時セット） | 種別判断の後押し。AVIFを入れるなら価値が出る |

### 2-5. 非機能
| 要素 | 判定 | 理由 |
|---|---|---|
| プライバシー表記（画像を送信しない明記） | **必須v1** | 最大の差別化メッセージ。ファーストビューに明示 |
| アクセシビリティ（キーボード操作・D&D代替・WCAG AA） | **必須v1** | 黒田さんのWCAG重視スタックの体現＝公開物の自己説明。ファイル選択ボタン＝D&D代替を必ず用意 |
| レスポンシブ | **必須v1** | clamp標準スタック。スマホでも崩れない |
| 対応ブラウザ | **必須v1（最新Chrome/Edge/Safari/Firefox）** | WASM・Web Worker前提。**非対応時のフォールバックメッセージ**を出す |
| エラー処理（非対応形式/破損/巨大画像） | **必須v1** | 1枚の失敗で全体が止まらない（スキップ＋該当のみエラー表示） |
| オフライン動作（PWA化） | **【将来】** | WASMは一度読めばオフライン処理可だが、PWA化は付加価値。v1は不要 |

### 2-6. やらないこと（スコープ外＝YAGNI排除の明示）
| 要素 | 判定 | 理由 |
|---|---|---|
| 画像編集（トリミング/フィルタ/明るさ補正/加工） | **【不要】** | 「最適化に集中」の一線。編集ツールは別物・無限に肥大する |
| 画像生成AI / 背景透過AI | **【不要】** | スコープ外。WASM完結・無料の前提と矛盾 |
| アップロード保存・クラウド同期・共有リンク | **【不要】** | 「送信しない」プライバシー訴求と真っ向矛盾。DB不要の設計思想に反する |
| アカウント/ログイン | **【不要】** | サーバー不要設計。YAGNI |
| アニメGIF/動画の最適化 | **【不要】** | 動画は別領域。静止画最適化に絞る |
| URL逆引き・既存ページ画像の取得 | **【不要】** | 別ツール（メタ工房等）の領分。混ぜない |

## 3. 技術的制約が仕様を縛る点（WASMの限界）

| 制約 | 実態 | 仕様判断への効き方 |
|---|---|---|
| **HEICデコード** | ブラウザネイティブ不可。libheif系WASM追加が必要（バンドル増・重い） | → **HEIC入力をv1から外す**根拠。iPhone写真ニーズはあるが工数対効果でv1不要（§2-1）。将来別ロード（遅延import）で検討 |
| **AVIFエンコードの重さ** | WASMでも1枚あたり数百ms〜秒オーダー。大量一括でブラウザ凍結リスク | → AVIF採用なら**Web Worker＋逐次処理＋「重い」注記＋進捗バー**が必須セット。「入れる＝Web Worker必須」が連動条件 |
| **メモリ上限** | 巨大画像（数千万px）/大量同時デコードでWASMヒープが落ちる | → **入力ファイルサイズ上限＋枚数ガード＋逐次処理**が必須（§2-1）。「無制限」は謳わない |
| **メインスレッド凍結** | エンコードは同期的に重い。UIが固まると「壊れた」と誤認される | → **Web Worker化を任意でなく必須**に格上げ（§2-4）。進捗表示も必須 |
| **減色/量子化（imagequant）** | WASM追加でバンドル増 | → ④⑤⑥種別とPNG可逆を採るなら連動して導入。採らないなら不要 |
| **EXIF Orientation** | EXIF削除で回転情報が消え画像が倒れる | → 削除前にCanvasで正立焼き込みが必須（§2-3）。「EXIF削除＝向き正規化とセット」が技術的必然 |
| **ZIP生成** | 大量・大容量ZIPはメモリを食う | → 軽量ライブラリ（fflate等）＋ストリーミング寄りで。超大量は分割DL案内 |

> 結論: WASM完結は「サーバー不要・送信しない」という最大の武器の代償として「重い処理・メモリ・凍結」のリスクを抱える。**この制約が「HEICを外す」「AVIFはWorker必須」「枚数上限を設ける」という仕様判断を直接駆動している**。

## 4. DECISION LIST（黒田さんが壁打ちで決める論点＋Mado推奨）

| # | 論点 | Mado推奨 | 補足 |
|---|---|---|---|
| D1 | **HEIC入力**を対応するか | **v1は非対応（将来）** | デコード重・バンドル増。制作素材はPNG/JPEG主体で痛点に不要。「HEICは事前JPEG化を」案内で代替 |
| D2 | **自動種別推測**をv1で入れるか | **入れる。ただし「初期値・必ず手動で直せる・推測根拠を小さく表示」を厳守**し精度は追わない | 誤推測は信頼毀損。手動オーバーライド（必須）があれば推測ゼロでも痛点は解消する。工数を見て**最後に積む順**でよい |
| D3 | **EXIF/メタデータ削除をデフォルトON**にするか | **デフォルトON**（容量＋GPS等プライバシー）。Orientation正立焼き込みとセット | Web書き出し用途でEXIF不要。著作権情報を残したい例外はトグルで |
| D4 | **AVIF出力**をv1に含めるか（フル決定では含む） | **含める。ただしWeb Worker＋逐次＋「重い」注記を必須前提**に | 含めるなら凍結対策が連動必須。重ければ「WebP優先・AVIFは任意オン」UIに |
| D5 | **可逆/減色（④⑤⑥種別＋oxipng/imagequant）**をv1に含めるか | **含める（フル決定方針に整合）**。ただし**スクショ記事頻度が低いなら⑥は後回し可** | バンドル・工数増の主因。痛点核（①②③）はWebPだけで満たせる |
| D6 | **劣化確認UI**をどこまで作り込むか | **v1は「並置」必須＋「ルーペ」任意。スライダー比較は将来** | スライダー重ね比較は工数大。並置/ルーペで実用十分 |
| D7 | **元形式維持出力（JPEGのまま軽く＝MozJPEG相当）**を入れるか | **v1不要（WebP変換で代替）** | 「WebPにしたくない場面」が実務でどれだけ出るか次第。出るなら任意で |
| D8 | **入力枚数/サイズ上限のN値** | **一度に20〜30枚目安・1枚あたり〜25MB目安でガード**（実機で調整） | 無制限を謳わずフリーズを防ぐ。実装後に黒田さんの実画像で詰める★要確認 |
| D9 | **フォルダ一括/クリップボード貼付** | **フォルダ一括=任意（あれば便利）／クリップボード=不要** | 複数D&Dで実用は足りる |
| D10 | **出力ファイル名規則のデフォルト** | **元名維持＋拡張子のみ変換に追従**（連番は任意） | 元名維持が制作で迷わない |

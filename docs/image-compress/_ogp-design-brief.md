## 1. 配色方針（design-concept トークン準拠 / 独自色を作らない）

### 1-1. ★ライト基調 vs ダーク基調の判断 → **ダーク基調を推奨**

OGPはSNSタイムライン上で**周囲の白いカードUI（X / Facebook / Slack いずれも地が白〜淡灰）に対してコントラストで浮く**ことが映えの条件。本体 b-tech-cool はライトがデフォルトだが、**OGPに限ってはダーク基調（`--color-bg-base` ダーク値 `#03050a` 〜 `--color-bg-alt` ダーク値 `#0a0e14`）を採用**する。

理由（Haru判断）:
- タイムラインの白背景の中で、濃紺ベースのカードは縁取りなしでも輪郭が立ち、サムネとして埋もれない。
- 本体のフッターが「ライト/ダーク問わず常時ダーク固定」（tokens §6・`#0a0e14`）である通り、**b-tech-cool は濃色面を「ブランドの締め色」として正規に持っている**。OGPをこの締め色面で組むのは世界観として正当（独自色ではない）。
- アクセントのティールグリーン（`--color-accent` `#2ba89c` / ダーク値 `#5dd9cb`）が濃紺の上で最も鮮やかに発光する。ライト地だと accent が沈む。

> ★要確認1: ライト基調案も後述（§2 案C）で用意。黒田さんが「本体トップの第一印象はライトなので揃えたい」と考える場合はライト案も選べる。**Haru推奨はダーク基調**。

### 1-2. 使用するトークン値（ダーク基調採用時）

| 用途 | トークン / 実値 | 備考 |
|---|---|---|
| 背景ベース | `--color-bg-alt`(dark) `#0a0e14` ベタ、または `#03050a`→`#0a0e14` の微グラデ | フッター固定色と同系＝ブランドの締め色 |
| 主見出し（ツール名）文字 | `--color-text-base`(dark) `#e6edf6` | 高コントラスト白系 |
| サブコピー文字 | `--color-text-muted`(dark) `#a8b3be` | 主見出しより一段落ち着かせる |
| アクセント（ロゴ line2・バッジ・装飾ライン） | `--color-accent`(dark) `#5dd9cb`（hover `#7ae3d7` は使わない） | 点で効かせる。面で塗らない |
| バッジ地 | `--color-accent-bg`(dark) `rgb(93 217 203 / 12%)` ＋ 1px `--color-accent` 枠 | 「送信しない」バッジの低彩度地 |
| ブランドロゴ line1 | `#e6edf6`（白系） | 「Frontend」 |
| ブランドロゴ line2 | `--footer-logo-accent` `#5dd9cb`（本体フッターと同一固定） | 「Tools」 |
| 区切り/枠線 | `--color-border`(dark) `#232e40` | カード枠・細ライン |

- accent は **ロゴ line2・バッジ・細い天面ライン**だけに点で使う（design-concept §3-1「accent は点で効かせる・面で塗らない」を踏襲）。
- 独自の色は一切作らない。すべて tokens.css §1〜§3・§6 の実値。

## 2. レイアウト案（概念図 / 安全マージン込み）

全案共通の安全マージン: 1200×630 の**外周 80px は重要要素を置かない**（SNSサムネのトリミング・角丸マスク対策）。重要要素は中央寄りの **約 1040×470 のセーフエリア**内に収める（§4-4「重要要素は中央寄せ」）。

### 案A【Haru推奨】中央スタック型（ツール名主役・最も素直）

```
┌──────────────────────────────────────────────────────┐ 1200×630
│  ◇ Frontend / Tools（左上・小 / ブランド地続き）        │  ← 外周80pxマージン
│                                                        │
│                                                        │
│        ┌─ accentの細い天面ライン ─┐                     │
│                                                        │
│            画 像 仕 分 け 圧 縮 く ん                    │ ← 主見出し(中央/最大)
│              ─ 種類別に選べる画像圧縮 ─                  │ ← サブコピー(muted)
│                                                        │
│            [ 🔒 ブラウザ内完結・送信しない ]            │ ← accentバッジ(中央)
│                                                        │
│                                                        │
│                          kurodafolio.com/tools（右下・小）│ ← ドメイン(控えめ)
└──────────────────────────────────────────────────────┘
```

- **ねらい**: 「これは何か（ツール名）」を最大面積で即伝達。視線が中央に一点集中。タイムラインで最も読み取りやすい。
- **メリット**: 構図が単純で破綻しない。文字主役なので解像度・縮小に強い（SNSで縮んでも読める）。テンプレ化（§4）が容易＝他ツールにそのまま使い回せる。
- **デメリット**: ビジュアル要素（アイコン・モチーフ）が弱く、やや真面目。差別化要素（安心バッジ）が3番手の視線順になる。
- 安全マージン: 主見出し・サブ・バッジすべて中央セーフエリア内。ロゴ/ドメインは隅だが「欠けても致命傷にならない補助要素」なので隅でよい。

### 案B 左右2分割型（左コピー・右ビジュアル / 機能の手触りを見せる）

```
┌──────────────────────────────────────────────────────┐ 1200×630
│  ◇ Frontend / Tools                                   │
│                                  ┌─────────────────┐   │
│   画像仕分け圧縮くん               │  ▢ before        │   │ ← 右:圧縮モチーフ
│   ─────────────                  │     ↓ 圧縮        │   │  (画像カード+矢印+
│   種類別に選べる画像圧縮           │  ▢ after (小)     │   │   サイズ縮小の図)
│                                  │  例: 2.4MB→380KB │   │
│   [ 🔒 送信しない ]               └─────────────────┘   │
│                                                        │
│   kurodafolio.com/tools                                │
└──────────────────────────────────────────────────────┘
```

- **ねらい**: 「何ができるか（圧縮でサイズが小さくなる）」を**図で**直感的に見せる。before→after のサイズ縮小モチーフが機能を語る。
- **メリット**: ツールの便益（軽くなる）が視覚的に伝わり、テキストを読まなくても用途が分かる。情報量が多く「ちゃんとした道具」感が出る。
- **デメリット**: 要素が増え、縮小時に右ビジュアルが潰れて読めなくなるリスク。安全マージンの制約で右カードが窮屈になりやすい。テンプレ化時に各ツールごとに右ビジュアルを描き起こす手間が出る（再利用性が案Aより低い）。
- 安全マージン: 右ビジュアルを外周80pxに寄せすぎないこと。左コピー群はセーフエリア左半分に収める。

### 案C ライト基調・横帯型（本体トップの第一印象に寄せる / ★要確認1の対案）

```
┌──────────────────────────────────────────────────────┐ 1200×630（地=白/淡灰 #f4f6f8）
│  ◇ Frontend / Tools                                   │
│                                                        │
│        画像仕分け圧縮くん                               │ ← 濃紺#1a2b3c文字
│        種類別に選べる画像圧縮                            │ ← muted #586674
│                                                        │
│   ━━━━━ accent帯(teal #2ba89c) ━━━━━━━━━━━━━━━━━     │ ← 下1/3にaccent面
│   [ 🔒 ブラウザ内完結・送信しない ]   kurodafolio.com   │ ← 帯の上に白文字バッジ
└──────────────────────────────────────────────────────┘
```

- **ねらい**: 本体トップ（ライトデフォルト）と第一印象を完全統一。清潔・誠実トーンを前面に。
- **メリット**: 本体サイトを知る人に「同じところの道具だ」と最短で伝わる。accent帯が下部のアンカーになり安定。
- **デメリット**: SNSの白背景タイムラインで地（白〜淡灰）が同化し、輪郭が立ちにくい（§1-1 の懸念）。映えは案A/Bに劣る。
- 安全マージン: accent帯は全幅でよいが、帯**上の文字**は左右80pxマージンを守る。

### 推奨まとめ

**第一推奨＝案A（中央スタック型・ダーク基調）**。理由: (1) SNSタイムラインでの視認性・映えが最も高い、(2) 文字主役で縮小に強い、(3) 将来の他ツールOGPへテンプレ流用が最も容易（§4）。案Bは「機能を図で見せたい」を優先する場合の対案、案Cは「本体ライトと第一印象を揃えたい」場合の対案。

## 3. タイポgrafィ方針（tokens 準拠 / 1200×630 で視認できるサイズ）

OGPは固定ピクセル画像なので rem ではなく **px実寸**で設計する（トークンの比率・フォント指定は踏襲）。

| 要素 | フォント | ウェイト | サイズ感(1200×630基準) | 色 |
|---|---|---|---|---|
| ツール名（主見出し） | `--font-family-sans` Noto Sans JP（和文主役） | `--fw-bold` 700 | **約 88〜104px**（画面幅の約 1/12〜画数で調整・1行で収める） | text-base |
| サブコピー（キャッチ） | Noto Sans JP | `--fw-regular` 400〜`--fw-medium` 500 | 約 34〜40px | text-muted |
| 「送信しない」バッジ文字 | Noto Sans JP | `--fw-medium` 500 | 約 28〜32px | accent or text-base |
| ブランドロゴ「Frontend / Tools」 | **`--font-family-en` Inter**（本体ロゴ実値・§1-7a） | `--fw-bold` 700 | 約 26〜30px（2行・line-height 1.15） | line1=white系 / line2=accent |
| ドメイン表記 | `--font-family-mono` JetBrains Mono | 400 | 約 22〜24px | text-muted |

- **和文主役はNoto Sans JP**（ツール名「画像仕分け圧縮くん」は和文なので Inter ではなく Noto）。ロゴの「Frontend / Tools」だけ英字なので Inter（本体ロゴと同じ font-heading）。これは design-concept §1-3 のフォント三役分担に忠実。
- 字間: 見出しは `--ls-tight -0.01em` 相当で締める。ドメインは `--ls-wide` で少し開ける（本体 © と同じトーン）。
- サイズ根拠: 主見出しは「SNSで最小表示（X タイムライン上 ≒ 横 500px相当）に縮んでも読める」逆算で px を決める。88px は 1200px 中で約7.3%＝縮小後も判読可能なライン。

## 4. ★制作手法の推奨 → **(a) HTML+CSSテンプレ→browserスクショ→PNG**

3候補のうち **(a) を強く推奨**。

| 手法 | 再利用性 | スタック相性 | 判定 |
|---|---|---|---|
| **(a) HTML+CSS で1200×630テンプレ→browser撮影→PNG** | ◎ tokens.css を直接 `@import`/リンクして流用。色変更ゼロで世界観統一 | ◎ Vite/handlebars/SCSS と同じHTML/CSS。学習コストなし | **推奨** |
| (b) SVG手組み | △ ベクタで綺麗だが、和文フォント埋め込み・tokens連携が手間 | △ SCSS変数を直接使えない | 非推奨 |
| (c) Figma等手作業 | ✗ デザインデータが別資産化し、トークン変更時に追従しない | ✗ コードベース外でブランド更新が二重管理に | 非推奨 |

### 推奨理由（再利用性 / スタック相性）

1. **トークン直結でブランドが自動同期**: `tokens.css` を `<link>` するだけで b-tech-cool の色・フォント・余白が1:1で効く。本体トークンを更新したらOGPも追従＝二重管理が消える（design-concept の「設計言語＝保守性の土台」思想と一致）。
2. **黒田さんのスタックそのまま**: HTML+CSS は Vite/handlebars/SCSS と完全に同じ技術。新しいツールも、学習コストなしで誰でも文言だけ差し替えられる。
3. **再現性のある生成**: `1200×630` の固定 viewport で Playwright/Puppeteer 等のheadless browserにスクショさせれば、毎回同一ピクセルで再生成できる（手作業のズレが出ない）。CIにも乗せられる。

### ★テンプレ化の青写真（将来ツールが増える前提）

```
docs/_portal/design/ogp-template/
  ├─ ogp.html          … 1200×630固定の枠。tokens.cssを<link>。
  │                       data属性で {tool-name} {tagline} {badge} {domain} を差し替え
  ├─ ogp.css           … OGP専用レイアウトCSS（safe-area 80px / 中央スタック=案A）
  └─ generate.md       … headless撮影手順（viewport 1200×630・deviceScaleFactor 2 → PNG）
```

- 案Aを基本テンプレにすれば、新ツールは **ツール名・キャッチ・カテゴリ色だけ差し替えて**同じ構図でOGPを量産できる。
- 出力は `{tool}/ogp.png`（各ツール同階層）。本ツールは `tools/image-compress/ogp.png`。
- 実装担当: テンプレHTML/CSS自体は Haru が概念モック相当で起こせるが、**headless撮影→PNG書き出しの本番運用（Vite/scriptビルド組み込み）は Cody 範囲**（役割境界：Haru は設計言語まで）。

## 6. ブランド表記（Frontend Tools / kurodafolio の切り分け）

ハンドオーバー方針（画面ブランド＝Frontend Tools / 本体＝kurodafolio）をOGPでどう扱うか。

| 表記 | OGPでの扱い | 根拠 |
|---|---|---|
| **「Frontend / Tools」ロゴ** | **入れる**（左上・小・2行・Inter・line2=accent #5dd9cb） | 画面ブランド。本体ヘッダー/フッターのロゴと同マークアップ思想（design-concept §1-7a / Phase2確定）。OGPでも「どの道具棚の道具か」を示す |
| **`kurodafolio.com/tools`（ドメイン）** | **入れる**（右下・小・mono・muted） | 本体ドメイン認知を育てる（§4-1/§2-2 のドメインパワー育成方針と一致）。OGメタの og:site_name=kurodafolio とも整合 |
| 「kurodafolio」社名/個人名 | **OGP画像内には出さない** | og:site_name・JSON-LD authorで機械可読に担保済み。画像内に出すと情報過多。ドメイン表記で足りる |

> 切り分けの考え方: **ロゴ＝ブランド（Frontend Tools）／ドメイン＝帰属（kurodafolio.com）**。両方を「隅に小さく」置き、中央セーフエリアは必須3要素に専念させる。ロゴ・ドメインは安全マージンの都合で隅に置くが、欠けても致命傷にならない補助情報なので隅でよい（中央寄せ必須は必須3要素＝ツール名・キャッチ・バッジ）。

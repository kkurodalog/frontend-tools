## 0. 結論（先出し）

**推奨: 折衷案 ＝「スクロール特化で v1 を立て、内部設計とブランド名だけ汎用化に耐える形にしておく」**

- **看板（ツール名）**: スクロール特化のまま。SEO・分かりやすさで最強。
- **サブドメイン**: 特化語ではなく**中立語**にして将来の引っ越しコストをゼロにする。
- **内部設計**: 仕様書の single source of truth / プリセットの器を、最初からアニメカテゴリを増やせる構造で実装。
- **横展開ルート**: 将来は「1つの汎用ツールに統合」ではなく**「カテゴリ追加で1ツールを育てる」**。別サブドメイン乱立はしない。

理由を一言で:「汎用キーワードは競合が強大で後発に勝ち目が薄い」「特化語は端境期で空きがある」「"汎用の器"は名前を汎用にしなくても**設計**で確保できる」——この3点で、看板は特化・中身は汎用、が最も損が少ない。

## 1. 検索流入（SEO）の観点 — Web検索で裏取り

### 1-1. 特化キーワード「スクロール アニメーション ジェネレーター」系

**実測: 端境期だが、既に強い専業の先行者が1ついる。**

- `digrart.jp/tools/css-scroll-animation-generator/`（CSSスクロールアニメーション ジェネレーター｜JS不要）が、まさに本ツールと同じ土俵に存在。**フェードイン/拡大/スライドイン/回転/ぼかし等10種プリセット＋スライダー微調整＋animation-timeline（JS不要）＋コピペ出力**と、機能面でかなり作り込まれている。特化キーワードの上位は**既に1社に押さえられ始めている**。
- ただし scroll-driven animations 自体が **2026時点で端境期**（Chrome/Edge対応・Safari道半ば・Firefox未対応）であり、解説記事（MDN/ICS/Qiita/とほほ）は厚いが、**ツールの母数はまだ少ない**。汎用ジャンルほど飽和していない＝後発でも切り込む余地は残る。

### 1-2. 汎用キーワード「CSS アニメーション ジェネレーター」系

**実測: 完全に飽和。後発の黒田さんが上位を取るのは非現実的。**

- on-ze.com / Toolpods / bfotool / homepage.co.jp（CSS ANIMATION GENERATOR）/ webcode.tools / **Animista**（事実上の定番）/ KASOU・CodeCampus等のまとめ記事——と、**老舗・被リンク蓄積・知名度のある競合がずらり**。Animista級のドメインと正面から汎用語で殴り合うのは、後発の個人サイトでは消耗戦になる。
- 「特化サイトはSEOで勝ちやすい」の一般原則は本件にも当てはまる：**汎用語＝強豪レッドオーシャン、特化語＝先行者1社はいるが端境期でまだ薄い**。汎用名で立てると、最も検索ボリュームのある汎用語で勝てず、かつ特化語の訴求も薄まる「どっちつかず」になる。

### 1-3. 本当の空き地 ＝「AIプロンプト出力」軸（最重要発見）

- 「AI プロンプト × アニメーション実装 × Cursor/v0/初心者」で検索しても、**この機能を持つジェネレーターは1つも出てこない**。Cursor/v0解説記事ばかりで、「GUIで選んでAIへの実装指示文を吐くツール」は**国内外で直接競合が見当たらない＝完全な空き地**。
- 示唆: **v1の勝ち筋はそもそも「スクロール vs 汎用」の軸ではなく「AIプロンプト出力」という第3の軸**。看板を特化にしても汎用にしても、digrart や Animista とコード生成の質だけで戦えば不利。**"AIプロンプトも出せる"を前面化すること自体が、特化/汎用の選択より検索流入インパクトが大きい**（仕様書 §6 の差別化と完全一致）。

**SEO結論**: 汎用語は勝てない（飽和）。特化語は端境期で取りに行ける（先行1社あるが薄い）。最大の差別化は特化/汎用ではなく「AIプロンプト」軸。→ **看板は特化語で取りに行き、AIプロンプト軸を必ず併記する**のが検索流入最適。

## 2. 利便性・ユーザー体験の観点

- **主ターゲット P1（AIで作る初学者）にとって「自分の道具だ」と認識しやすいのは特化**。「スクロールで“ふわっと出す”をやりたい」と思って来た人に、「スクロール演出ジェネレーター」は**用途がそのまま名前**で刺さる。汎用名（"アニメーションジェネレーター"）は守備範囲が広い反面、「結局これで“あの動き”ができるの?」と入口で迷う。初学者は**広さより具体性**で安心する。
- **汎用型はUI複雑化リスクが大きい**。仕様書 §2-2 は「パラメータは多すぎないことが正義／無関係なパラメータは隠す（プログレッシブ・ディスクロージャー）」を初学者保護の核に据えている。汎用の器を最初から見せると、ホバー/ローディング/テキスト演出…とカテゴリ選択が増え、**初学者が"自分に関係ない選択肢"の森で迷子**になる。v1で見せる選択肢は絞るほど良い。
- 結論: **v1の見た目・導線は特化が正解**。「スクロール演出だけやる潔い道具」が初学者の体験を最大化する。汎用化しても、UI上は「カテゴリで切り替える」形にして、初学者には1カテゴリだけ見せれば複雑化は回避できる（→ §3）。

## 3. 拡張性・アーキテクチャの観点

### 3-1. 特化で始めても内部を汎用拡張に耐える形にできるか → **できる**

- 仕様書 §5-2-C の **single source of truth（1状態オブジェクト → コード/プロンプト/プレビューの3出力が派生）** は、**プリセット定義をデータとして外出しできる構造**を意味する。プリセットを `{ id, category, params定義, codeTemplate, promptTemplate, previewMode }` のような**プリセット記述子の配列**として持てば、スクロール5本も、将来のホバー/ローディング系も**同じ器に行を足すだけ**で増やせる。
- つまり「スクロール特化」は**UIの見せ方とコピーの問題**であって、**アーキテクチャを特化に固める必要はない**。最初から `category: "scroll"` を1つ持つ汎用器として実装し、v1ではUI上 scroll カテゴリのみ露出すればよい。**後から汎用化してもアーキテクチャ的に無理がない**（むしろ仕様書設計はそれを前提にしている）。
- 設計上の要件（軽微）: ①プリセットをカテゴリ属性付きデータ駆動にする ②URLシリアライズに将来 `category` を含められる予約を入れておく ③プロンプトテンプレの3ブロック定型をカテゴリ非依存にする。この3点を v1 実装時に守れば、汎用化の追加コストは「カテゴリを足す＝プリセット記述子を増やす」だけになる。

### 3-2. 横展開ルート — 別サブドメイン乱立 vs 1ツール統合

`{tool}.kurodafolio.com`（1サブドメイン=1ツール）構造を踏まえた2ルート比較:

| 観点 | A. 別サブドメインで横展開（scroll版/hover版/loading版…） | B. 1つの汎用ツールにカテゴリ追加で統合 |
|---|---|---|
| 運用 | ツール数だけビルド・保守・更新が増殖。個人事業では重い | 1コードベースを育てるだけ。保守集中 |
| SEO | サブドメインごとにドメイン評価がゼロから。被リンク・流入が分散 | 1ドメインに被リンク・流入・回遊が集約され評価が育つ |
| ブランド | 「黒田さんのツール群」が散らばり、相互送客しにくい | 「動きを作るならここ」の1拠点に集約。記事連動も一箇所 |
| 初学者UX | ツールを探し回らせる | 1箇所で全部選べる（ただしUIは§2の通りカテゴリで絞って見せる） |

→ **B（1ツールをカテゴリ追加で育てる）が運用・SEO・ブランドすべてで有利**。別サブドメイン乱立は個人事業の保守負荷とドメイン評価分散で不利。**「将来アニメ全般」は新サブドメインを増やす形ではなく、この1ツールにカテゴリを足す形で実現するのが正解**。これが §0 で「統合より育成」を選んだ根拠。

## 4. ブランド・リブランドリスクの観点

「特化看板 → 後で汎用化」のコストを具体評価する。

- **最大のリスクは"看板名の変更"ではなく"サブドメイン（URL）の引っ越し"**。被リンク・検索評価・ブックマーク・記事内CTAは**URLに紐づく**。サブドメインを `scroll.kurodafolio.com` のような特化語にしてしまうと、汎用化時に `animation.kurodafolio.com` 等へ引っ越したくなり、**301リダイレクト運用・被リンク資産の一部毀損・記事内リンク全張り替え**が発生する。これは実コスト。
- **ツール名（看板テキスト）の変更コストは相対的に軽い**。h1・metaタイトル・OGの差し替えで済み、URLが変わらなければSEO資産はほぼ温存できる。**「スクロール演出ジェネレーター → ◯◯（汎用名）」へのリネーム自体は低コスト**。
- **回避策（折衷の肝）**: **看板テキストは特化語にするが、サブドメインは中立語にしておく**。例 `motion.kurodafolio.com` や `anime.kurodafolio.com`。こうすると：
  - v1: URLは中立、表示名「スクロール演出ジェネレーター」で特化SEOを取りに行く
  - 汎用化時: **URLは据え置き**（引っ越しゼロ＝被リンク毀損ゼロ）、表示名だけ汎用名へリネーム
  - → **リブランドの実コストの大半（URL引っ越し）を最初から消せる**。「最初から汎用名にしておけばリブランド不要」論のメリットを、**看板まで汎用にしなくても、サブドメインの中立化だけで8割回収できる**。
- 「最初から汎用名」案（案Y）の弱点: §1（汎用語で勝てない）§2（初学者が迷う）を v1 から背負う。リブランド回避という1点のためにSEO・UXの主戦力を捨てるのは割に合わない。**サブドメイン中立化で同じ回避効果が得られる以上、看板まで汎用にする必要はない**。

## 5. 結論：v1の推奨と命名方針

### 5-1. 推奨 = 折衷（特化で立て、中身とURLを汎用耐性にする）

| 項目 | 推奨 | 根拠 |
|---|---|---|
| 看板（表示名） | **スクロール特化のまま** | §1 特化SEOを取れる端境期 / §2 初学者に刺さる具体性 |
| サブドメイン（URL） | **中立語**（特化語にしない） | §4 将来の汎用化で URL 引っ越し＝被リンク毀損を回避 |
| 内部設計 | **最初から汎用の器**（プリセットをカテゴリ属性付きデータ駆動） | §3-1 後付け汎用化のコストを「カテゴリ追加」だけに圧縮 |
| 横展開 | **別サブドメイン乱立せず、1ツールにカテゴリ追加で育てる** | §3-2 運用/SEO/ブランドで有利 |
| 前面に出す軸 | **「AIプロンプトも出せる」を看板サブタイトルで常時主張** | §1-3 特化/汎用より大きい唯一の空き地 |

→ **案X（特化）を基本としつつ、案Yの唯一の利点（リブランド回避）をサブドメイン中立化で吸収する折衷**。SEO・UXの主戦力を捨てずにリブランドリスクだけ消す、いいとこ取り。

### 5-2. 命名方針（具体案）

- **看板（h1 / metaタイトル）**: `スクロール演出ジェネレーター`（仕様書 §9 推しトップ1。SEO最強・分かりやすさ最強）
- **サブタイトル**: `— コードもAIプロンプトも出せる`（§1-3 の唯一の空き地＝AI軸を検索/訴求の両面で常時前面化）
- **サブドメイン**: **中立語推奨**。候補:
  - `motion.kurodafolio.com`（汎用化後も "Motion ＝ 動き全般" でそのまま使える / 第一推奨）
  - `anime.kurodafolio.com`（短い・覚えやすいが "anime" の誤読ニュアンス★）
  - `move.kurodafolio.com`（短く中立 / やや無機質）
  - → **`motion` 推奨**。「動きを作る拠点」という将来の汎用ブランドにそのまま昇格できる。**特化語の `scroll.` は避ける**（引っ越しの種になる）。
- **将来の汎用化時のリネーム例**: 表示名のみ `モーションジェネレーター` / `Webアニメーション作成ツール` 等へ差し替え。**URL（`motion.kurodafolio.com`）は据え置きでSEO資産を継承**。スクロールは「第1カテゴリ」として中に残す。

### 5-3. 黒田さんの確定ゴール（将来アニメ全般化）への接続

この折衷ルートは黒田さんの確定ゴールに無理なく接続する。**v1は「スクロール演出ジェネレーター（motion.kurodafolio.com）」という潔い特化の顔で出し、特化SEOと初学者への分かりやすさを最大限取りに行く。一方その中身は、プリセットをカテゴリ属性付きデータとして持つ汎用の器として実装し、URLも中立語にしておく。** これにより、将来ホバー・ローディング・テキスト演出…を足すときは「別ツールを新規に立てる」のではなく「同じ器にカテゴリを増やし、表示名だけ汎用名へリネームする」だけで済む。被リンク・流入・記事連動の資産を一切毀損せず、特化スタートのSEO/UX上の利得を保ったまま、シームレスに「Webの動きを作る拠点」へと育てられる。

## 検証済み

- **検証済み: SEOの裏取り（推測でなくWeb検索）** — 特化語は専業先行者 digrart.jp が機能充実で存在するが scroll-driven 自体が端境期でツール母数は薄い、汎用語は Animista/on-ze/Toolpods 等で完全飽和、を実検索で確認。さらに「AIプロンプト×アニメ実装」軸は直接競合ゼロ＝最大の空き地、という第3の発見を得た。特化/汎用どちらが上位を取りやすいかを競合強度ベースで比較した。
- **検証済み: リブランド/URL引っ越しコストの具体評価** — §4 で「コストの本体はツール名変更ではなくサブドメイン(URL)引っ越しによる被リンク・検索評価の毀損」と特定し、回避策として「看板は特化・サブドメインは中立語」を提示。リブランド実コストの大半をv1時点で消せることを具体的に示した。
- **検証済み: 推奨が4観点の総合かつゴール接続** — 利便性(§2)/検索流入(§1)/拡張性(§3)/ブランドリスク(§4)を総合し折衷を導出。§5-3 で黒田さんの確定ゴール（将来アニメ全般化）へ「カテゴリ追加＋表示名リネーム＋URL据え置き」で接続することを明示した。
- **★要確認（黒田さん判断事項）**: (1)サブドメインを `motion`（第一推奨）でよいか／`anime`・`move` 等の代替を採るか — 一度決めると引っ越しコストが乗るため、汎用化後も使える語をここで確定したい。(2)看板サブタイトルに AI 軸（「コードもAIプロンプトも」）を常時出す方針でよいか（§1-3 の最大差別化を名前段階で出すか）。

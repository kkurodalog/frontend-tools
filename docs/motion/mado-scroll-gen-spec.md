## 0. 最重要・このツールの軸（読み飛ばし厳禁）

黒田さんが示した核心ポジショニング:

> AIでWebサイトを作るとき、「アニメーション系の動作をつけるプロンプトが難しい」ために、**フロントに詳しくない初学者がうまくアニメを作れない**。この問題を解消しつつ、ブログ記事への回遊も促したい。

→ よって本ツールは **単なるCSSジェネレータではない**。
**「AIコーディング時代に、フロント初学者が"言語化できないアニメーション"を実装できるようにするツール」** と定義する。
出力モードに **(b) AIプロンプト生成** を据えることが、海外の純粋コード生成ツール（Fixie Tools / FWD Tools 等）に対する決定的差別化になる。**この軸を全機能の設計判断の最上位に置く。**

## 1. コンセプト / ポジショニング

### 1-1. 一言コンセプト

> **「動かしたい"あの感じ"を選ぶだけ。コードでも、AIへの指示文でも出せる、スクロール演出の翻訳機。」**

「ふわっと出てくる」「数字がパチパチ増える」といった、初学者が**言葉にできない動きを、GUIで選んで→そのまま使える成果物（コード or AIプロンプト）に翻訳する**ツール。

### 1-2. 「AI×初学者のアニメプロンプト課題」をどう体現するか

初学者がAI（Claude / Cursor / v0 等）にアニメを実装させられない理由は3つある。本ツールはこの3つを順に潰す:

| 初学者のつまずき | 本ツールの解消手段 |
|---|---|
| ① そもそも**何という動きか名前を知らない**（「フェードイン」「stagger」という語彙がない） | プリセットを**動きの見た目（プレビュー付き）で選べる**。語彙を知らなくても選択できる |
| ② 動きの細部（速さ・方向・発火位置）を**言語化できない** | パラメータをGUIスライダーで調整 → その設定値が**自動で日本語の指示文に変換**される |
| ③ AIに頼んでも**実務で動くコードが返らない**（ブラウザ対応・フォールバック欠落） | 「フォールバック込みで」「animation-timelineを優先しつつ非対応ブラウザではIntersectionObserverで」といった**実務要件を織り込んだプロンプト**を自動生成する |

→ つまり本ツールは「初学者のGUI操作」と「AIへの的確な発注」の間の**翻訳レイヤー**として機能する。これが新規性の核。

### 1-3. 想定ユーザーの具体像と利用シーン

| ペルソナ | 利用シーン |
|---|---|
| **P1: AIでサイトを作る初学者**（主ターゲット）。v0 / Cursor / Claude でLPを作っているが、アニメだけ思い通りにならない | 「ヒーローの見出しをふわっと出したい」→ プリセット選択 → **AIプロンプトをコピー** → Cursorに貼って実装完了 |
| **P2: フロント学習者**（ブログ読者と同層 / 駆け出し） | スクロール演出を**自分のコードで**実装したい → プリセット選択・調整 → **コード出力をコピー** → 仕組みは連動するブログ記事で学ぶ |
| **P3: 同業フリーランス / 制作者** | 案件で「ふわっと出る」演出を量産したい → 微調整して**実務で動くフォールバック込みコードを即取得**（時短ツールとして常用） |

主ターゲットは **P1（AIで作る初学者）**。ここが競合との差別化が最も効く層であり、黒田さんの課題意識の中心。P2/P3 は同時に取り込める副ターゲット。

## 2. コア機能（MVPに入れるもの）

### 2-1. アニメーションプリセット候補リスト

frontend-note `scroll-animation` シリーズ（5パターン）と1:1で対応させ、**記事⇄ツール回遊を最初から成立させる**。これがブランド資産との最強の相乗効果。

| # | プリセット | 対応する動き | 中核技術 | 連動記事(snippet) | v1採否 |
|:--:|---|---|---|---|:--:|
| 1 | **フェードイン / スライドイン** | ふわっと出現（上下左右4方向＋距離指定） | IntersectionObserver + CSS | 001 fade-in | **v1必須** |
| 2 | **順次表示（stagger）** | 複数要素が少しずつ遅れて連続表示 | IO + `transition-delay` 連番 | 002 stagger | **v1必須** |
| 3 | **スクロール進捗バー** | 読了プログレスバーが伸びる | `animation-timeline: scroll()` ＋JSフォールバック | 003 progress-bar | **v1必須** |
| 4 | **パララックス** | 背景/要素が視差でずれて奥行きが出る | `animation-timeline: view()` ＋フォールバック | 004 parallax | v1任意（§8） |
| 5 | **カウントアップ** | 数値がパチパチ増える | IO + `requestAnimationFrame` | 005 count-up | v1任意（§8） |

> 補足: 1〜3を「v1コア3本」、4〜5を「早期拡張」とする方針は §8 で詳述。シリーズ全5本と将来1:1で揃う設計を最初から持たせ、**拡張してもブレない器**にする。

### 2-2. プリセット別の調整パラメータ

全プリセット共通で「触りすぎて壊れない」よう、**初学者向けに既定値を最適化し、上級者向けに詳細を開く**二段構え（プログレッシブ・ディスクロージャー）。

| パラメータ | 適用プリセット | UI | 既定値の考え方 |
|---|---|---|---|
| **duration（再生時間）** | 全て | スライダー 0.1〜2.0s | 0.6s（自然に見える既定） |
| **delay（遅延）** | 1,2,4,5 | スライダー 0〜1.0s | 0s |
| **easing（緩急）** | 1,2,4 | プリセット選択（ease-out / ease-in-out / 直線 / ふわっと(cubic-bezier)） | ease-out（出現演出の定番） |
| **移動距離 / 方向** | 1（slide）,4 | 方向ボタン(↑↓←→)＋距離px | 上方向32px |
| **発火位置 threshold** | 1,2,5 | スライダー（要素が何%見えたら発火） | 15%（早すぎ/遅すぎ回避） |
| **繰り返し（once / every）** | 1,2,5 | トグル | once（1回だけ＝実務既定） |
| **stagger間隔** | 2 | スライダー 0.05〜0.3s | 0.1s |
| **開始値 / 終了値 / 桁区切り** | 5（count-up） | 数値入力＋カンマ区切りトグル | 0→100 / カンマあり |

> 設計方針: パラメータは**多すぎないことが正義**。初学者が迷子にならないよう、各プリセットで露出するのは上表の該当行のみ（無関係なパラメータは隠す）。

### 2-3. ライブプレビュー仕様

- 画面中央に**プレビュー領域**を常設。プリセット/パラメータを変えると**即座に再生**（debounce 200ms程度）。
- プレビュー内に「▼ スクロールして発火」を疑似再現する **[もう一度再生] ボタン**を置く（実スクロールせずに発火を確認できる）。スクロール連動系（3,4）はプレビュー枠内をミニスクロールできるサンドボックスにする。
- `prefers-reduced-motion` のプレビュートグルを用意し、**アクセシビリティ配慮を体験として見せる**（黒田さんの強みの体現＝§6差別化にも効く）。
- プレビューは**実際に出力されるコードと同一の実装**で描画する（プレビューと出力の乖離をゼロにする＝信頼性の核）。

## 3. 出力形式（このツールの肝・複数モード）

3つの出力タブを **同一画面の下部「出力パネル」に並列**する。中央プレビューは常時見えたまま、出力タブだけ切り替わる（§4 参照）。

```
[ コード ] [ AIプロンプト ] [ 設定を共有(URL) ]
```

### 3-1. (a) コード出力 — 実務でそのまま動く

**設計原則: native scroll-driven animations 優先 ＋ IntersectionObserver フォールバック込み。** 「最新CSSだけ吐いて実務で動かない」海外ツールへの差別化点。

出力構成:
1. **HTML**（対象要素にクラス/data属性を付けた最小マークアップ）
2. **CSS**（`@keyframes` ＋ `@supports (animation-timeline: view())` で native を優先適用）
3. **JS**（IntersectionObserver フォールバック。`@supports` で native 非対応時のみ発火）
4. ヘッダコメントに**ブラウザ対応の一言**と`prefers-reduced-motion`対応を含める

> 出現系（1,2,5）は「IntersectionObserver が主・native は任意」、スクロール量連動系（3,4）は「animation-timeline が主・JS がフォールバック」と、**プリセットごとに主従を最適化**する。シリーズ記事の技術選定（structure.md）と完全整合。

コード出力例（フェードイン・上方向・抜粋イメージ）:
```css
@media (prefers-reduced-motion: no-preference) {
  .fade-in { opacity: 0; transform: translateY(32px); transition: opacity .6s ease-out, transform .6s ease-out; }
  .fade-in.is-visible { opacity: 1; transform: translateY(0); }
}
```
```js
const io = new IntersectionObserver((entries)=>{
  entries.forEach(e=>{ if(e.isIntersecting){ e.target.classList.add('is-visible'); io.unobserve(e.target);} });
}, { threshold: 0.15 });
document.querySelectorAll('.fade-in').forEach(el=>io.observe(el));
```
> 上はイメージ。実コードは実装フェーズで Cody が確定し、**Reid のコードレビュー（evaluation-gate.md / snippets-free 同等の品質ゲート）を通す**。ツールが吐くコード＝対外公開コード扱いとし、品質を担保する。

### 3-2. (b) AIプロンプト出力 — 新規性の中核

GUIで選んだプリセット＋パラメータを、**そのままClaude / Cursor / v0 に貼れる日本語の実装指示文**に変換する。初学者の「言語化できない動き」を、ツールが代わりに言語化する。

#### プロンプト出力 実例1: フェードイン（上方向・初学者がv0/Cursorに貼る想定）

```
あなたはフロントエンドエンジニアです。以下のスクロールアニメーションを実装してください。

# 実装してほしい動き
- 対象要素が画面に入ってきたとき、下から少し上へ移動しながらフェードインさせる
- 移動距離: 32px（下→上）
- 再生時間: 0.6秒
- 緩急: ease-out（最初速く、最後ゆっくり止まる）
- 発火タイミング: 要素が15%見えたら
- 一度表示したら繰り返さない（再表示しない）

# 技術要件（実務で動くように）
- まず CSS の native scroll-driven animations（animation-timeline: view()）で実装を試みる
- それに対応していないブラウザ向けに、IntersectionObserver によるフォールバックも必ず付ける
- prefers-reduced-motion: reduce のユーザーにはアニメーションを無効化する
- 余計なライブラリ（GSAP等）は使わず、素のHTML/CSS/JSで完結させる

# 出力形式
HTML / CSS / JS を分けて、コピペで動く形で提示してください。
```

#### プロンプト出力 実例2: カウントアップ（数値アニメ・初学者がClaudeに貼る想定）

```
あなたはフロントエンドエンジニアです。以下の数値カウントアップアニメーションを実装してください。

# 実装してほしい動き
- 対象の数字が、画面に入ったとき 0 から 1500 までパチパチと増えていく
- 再生時間: 1.2秒
- 3桁区切りのカンマを表示する（例: 1,500）
- 発火タイミング: 要素が30%見えたら
- 一度カウントしたら繰り返さない

# 技術要件（実務で動くように）
- IntersectionObserver で発火を検知する
- 数値の増加は requestAnimationFrame でなめらかに行う（setIntervalは使わない）
- prefers-reduced-motion: reduce のユーザーには、アニメさせず最終値を即表示する
- 素のJavaScriptで完結させる（ライブラリ不要）

# 出力形式
HTML / CSS / JS を分けて、コピペで動く形で提示してください。
```

> ポイント: プロンプトは「①動きの自然言語化 ②実務技術要件 ③出力形式指定」の**3ブロック定型**で生成する。②の技術要件（フォールバック・reduced-motion・脱ライブラリ）が**初学者が自力では書けない部分**であり、ここを自動で織り込むのが本ツールの価値。テンプレに `prefers-reduced-motion` 配慮を必ず含めることで、ブランドの「アクセシビリティ重視」を初学者の成果物にまで波及させる。

### 3-3. (a)(b)を1画面で両立させる方法

- 中央プレビューは**共通の真実**（同じ設定が即反映）。出力パネルのタブを切り替えるだけで、同じ設定から「コード」「AIプロンプト」が両方取り出せる。
- **どちらが正解という押し付けをしない**。「自分で書く人はコード」「AIに任せる人はプロンプト」と、ユーザーの作り方に合わせて選べる二刀流が体験の肝。
- 各出力に **[コピー] ボタン**＋コピー完了トースト。AIプロンプトには「Cursor / Claude / v0 に貼ってください」の一言ガイドを添える。

### 4-2. モバイルレイアウト（縦積み）

```
ヘッダー
↓ ① プリセット選択（横スクロールのチップ or アコーディオン）
↓ ② プレビュー（画面幅いっぱい・上部固定 sticky 候補）
↓ ③ パラメータ調整（折りたたみセクション）

```

### 4-3. 初学者がつまずかない導線設計

1. **入口で迷わせない**: プリセット選択は**プレビュー付きカード**。語彙を知らなくても「見た目」で選べる（§1-2 つまずき①対策）。
2. **触れば即わかる**: 何か変えると中央プレビューが即動く → 因果が直感的。
3. **完成の出口を明示**: 出力パネルに「自分で書くなら[コード] / AIに任せるなら[AIプロンプト]」と二択を明示。初学者は迷わず[AIプロンプト]へ誘導できる。
4. **詰まったら学べる**: 各プリセット下に「この動きを記事で学ぶ→」を常設し、ツールで詰まった瞬間にブログへ送る（§7 回遊）。
5. **失敗しても戻れる**: [既定に戻す]で常にリセット可能。壊して学べる安心感。

## 5. 技術土俵

### 5-1. 静的・ブラウザ内完結で全機能が成立するか → **成立する**

| 機能 | 実現方式 | 静的完結 |
|---|---|---|
| プレビューエンジン | 出力と同一のHTML/CSS/JSをiframe/sandboxで描画 | ✅ |
| コード生成 | テンプレート文字列にパラメータを差し込み | ✅ |
| AIプロンプト生成 | 定型3ブロックテンプレにパラメータを日本語で差し込み | ✅ |
| 設定の共有 | パラメータをURLクエリ/ハッシュにシリアライズ（サーバー不要） | ✅ |

→ サーバー・API一切不要。kurodafolio 静的基盤（Vite + Handlebars + FLOCSS）と完全整合。コスト0・プライバシー上の懸念もなし。

### 5-2. 実装上の技術的山場・注意点

| 山場 | 内容・対策 |
|---|---|
| **A. scroll-driven animations のブラウザ対応** | `animation-timeline` は Chrome/Edge 等 Chromium系は対応、Safari は道半ば・Firefox はフラグ/部分対応の**端境期**（2026時点）。→ **必ず IntersectionObserver フォールバックを同梱**。`@supports (animation-timeline: view())` で分岐。ツールの売りそのものなので、ここの設計が品質の生命線 |
| **B. プレビューと出力の一致** | プレビューを「それっぽい別実装」で作ると、出力コードを貼った結果と食い違い信頼を失う。→ **プレビューは生成コードと同一物をiframe sandboxで実行**して描画する設計を最初から取る |
| **C. パラメータ→コード/プロンプトの単一ソース化** | コード生成とプロンプト生成で別々にパラメータを解釈すると不整合が出る。→ パラメータを**1つの状態オブジェクト**に集約し、コード/プロンプト/プレビューの3出力がそこから派生する設計（single source of truth） |
| **D. reduced-motion の徹底** | 全出力に `prefers-reduced-motion` 分岐を必ず入れる。ツール自体のプレビューも尊重する。アクセシビリティはブランド差別化なので妥協しない |
| **E. プレビュー sandbox のセキュリティ** | iframe は `sandbox` 属性付き・同一生成コードのみ注入（ユーザー任意コード実行はさせない＝XSS面なし） |

> ★要確認: プレビューを「iframe sandboxで生成コードを実行」する方式は理想だが、実装コストが上がる。MVPでは**主要プリセット（出現系）はDOM直描画で簡易プレビュー**とし、スクロール連動系のみ sandbox とする折衷も可。どこまで厳密にやるかは実装フェーズで Cody と相談。

## 6. 競合との差別化（再確認）

海外ツール（Fixie Tools / FWD Tools 等＝純粋なコード生成・英語・最新CSS前提）に対し:

| 差別化軸 | 本ツール | 海外競合 |
|---|---|---|
| **AIプロンプト出力** | ✅ あり（新規性の核・初学者の発注を代行） | ❌ なし（コードのみ） |
| **日本語** | ✅ UI・プロンプト・解説すべて日本語 | ❌ 英語 |
| **フォールバック込み実用コード** | ✅ native＋IObserver＋reduced-motion 同梱 | △ 最新CSSのみで実務に落としにくいものが多い |
| **記事連動** | ✅ frontend-note scroll-animation 5本と1:1で回遊 | ❌ 解説資産なし |
| **アクセシビリティ配慮** | ✅ reduced-motion をプレビュー/出力に標準装備 | △ ツール依存 |

→ 勝ち筋は **「AIで作る日本人初学者」という、海外ツールが取りこぼしている層に、AIプロンプト出力＋日本語＋実用コード＋記事という4点セットで刺さること**。コード生成だけなら海外に勝てないが、「AI時代の初学者の翻訳機」というポジションは空いている。

## 8. MVPスコープ vs 将来拡張

### 8-1. v1 MVP（最小公開）— 「絞って早く出す」

| 含める | 切る / 後回し |
|---|---|
| **プリセット3本**: ①フェードイン/スライドイン ②順次表示(stagger) ③スクロール進捗バー | プリセット④パララックス ⑤カウントアップ（早期拡張で追加） |
| **コード出力（a）**: native＋IObserverフォールバック＋reduced-motion 込み | — |
| **AIプロンプト出力（b）**: ★**v1から必ず入れる**（これがツールの新規性＝外すと"ただのジェネレータ"になる） | — |
| ライブプレビュー（出現系はDOM直描画 / 進捗バーはミニsandbox） | iframe完全sandboxの厳密化（拡張で精緻化） |
| 設定共有URL | — |
| 関連記事ブロック（3本ぶん） | — |

**v1の設計判断:**
- **プリセットは3本に絞る**。出現系2本（最頻出・需要が太い）＋スクロール量連動1本（progress＝native scroll-driven animations の代表作で技術的見せ場）で、**「出現演出」と「スクロール連動」の両カテゴリを最小本数でカバー**。あれもこれも入れない。
- **AIプロンプト出力は v1必須**。プリセットを減らしてでもこれは入れる。これを後回しにすると競合と同質化し、公開する意味が薄れる。**「プリセット数 < プロンプト出力」が優先順位**。
- パララックス(④)は**プレビュー実装が最重**（視差の sandbox 描画が難物）なので v1から外し、早期拡張の筆頭に。カウントアップ(⑤)は実装は軽いが、出現系3本で器の検証を終えてから足す。

→ **「3プリセット ＋ コード/AIプロンプト二刀流」で最小公開**。シリーズ5本と将来1:1で揃う器を最初から設計しておく。

### 8-2. 将来拡張候補

| 拡張 | 内容 |
|---|---|
| プリセット ④パララックス ⑤カウントアップ 追加（→シリーズ5本と1:1完成） |

| Tailwind / Sass 形式でのコード出力 |
| プリセットの**お気に入り保存**（localStorage） |

## 10. 工数感の目安（S/M/L）

> 正式見積りは Mane が後で実施。以下は粒度感のみ。

| スコープ | 目安 | 山場 |
|---|---|---|
| **v1 MVP**（3プリセット＋コード/AIプロンプト二刀流＋簡易プレビュー＋記事連動） | **M（1〜2週相当）** | プレビューと出力の一致（§5-2 B）／single source of truth設計（§5-2 C）／プロンプトテンプレ設計 |
| **フル**（5プリセット＋sandbox精緻化＋組み合わせ＋ツール別プロンプト＋出力形式拡張） | **L** | パララックスのsandboxプレビュー／組み合わせプレビュー |

- プリセット1本を厳密sandboxプレビューまでやるか簡易DOMで済ますかで工数が振れる。**v1は簡易寄せでMに収める**のが現実的。
- AIプロンプト出力は**テンプレ設計さえ決まれば実装は軽い**（文字列組み立て）。新規性の割に低コストで載る＝費用対効果が高く、v1必須にしても工数を圧迫しない。
